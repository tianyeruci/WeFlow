# 邀请统计增量扫描系统消息遗漏排查执行计划（2026-05-23）

## 背景

服务端增量扫描完成后，符合条件的数据（被邀请或退出的系统消息）存在遗漏。目标是找到遗漏原因，给出可行方案，并把当前查询流程和逻辑记录到本地。

## 关键假设

- 本次先以排查、记录和方案设计为主，除非证据充分且改动边界很小，否则不直接修改业务行为。
- “服务端增量扫描”主要涉及 `WeFlow-main` 本地客户端扫描与 `WeFlow-Web` 服务端/远程同步接口之间的数据链路，需要同时查看两侧代码与已有文档。
- “被邀请或退出的系统消息”应对应微信/WeChat 群系统消息中的邀请入群、退群、移出群等文本解析逻辑，最终会落入邀请统计相关表或远端 raw events。
- 如果发现需要改公共表结构、同步协议、历史数据修复或大范围迁移，会集中列出问题与方案，先向用户确认。

## 成功标准

- 记录当前增量扫描从本地查询、过滤、解析、分页/游标、同步上传到服务端入库的详细流程。
- 明确列出可能导致遗漏的原因，并用代码路径、SQL/查询条件、已有文档或可复现推理支撑。
- 给出至少一个可落地修复方案，说明影响面、风险和验证方式。
- 本地保存完整排查记录与本次文件变更清单。

## 执行步骤

1. 定位增量扫描与邀请统计链路
   - 查看 `WeFlow-main` 中 `inviteStatsService`、`inviteStatsSyncService`、WCDB 查询服务及相关 IPC 调用。
   - 查看 `WeFlow-Web` 中 `/api/invite/sync`、`sync-request`、`invite-sync`、`invite-data` 等服务端接口。
   - verify: 列出核心文件、关键函数、调用顺序。

2. 阅读已有上下文与历史修复记录
   - 重点阅读 `docs/invite-scan-sync-watermark-plan-2026-05-22.md`、`docs/invite-scan-sync-watermark-change-log-2026-05-22.md`、相关 Supabase SQL。
   - verify: 提取现有水位、批次、raw events、解析和同步策略。

3. 梳理当前查询详细流程
   - 记录本地数据库查询条件：表名、字段、时间/sequence/msgId 水位、msgType/subType、群过滤、limit/order。
   - 记录服务端接收与入库逻辑：去重键、冲突策略、状态字段、游标更新时机。
   - 记录解析逻辑：如何判断“被邀请”和“退出/移出”系统消息。
   - verify: 生成 `docs/邀请统计增量扫描系统消息查询流程记录-2026-05-23.md`。

4. 排查遗漏原因
   - 检查水位推进是否可能先跳过后解析失败或后续补数据。
   - 检查分页排序字段是否稳定，是否存在同时间戳/同 sequence 丢页。
   - 检查系统消息类型或内容过滤是否过窄。
   - 检查本地查询与服务端去重口径是否不一致。
   - 检查增量扫描完成状态是否在部分群/部分页失败时仍被标记完成。
   - verify: 形成原因候选清单，标注证据强弱。

5. 如有条件，执行最小验证
   - 优先运行静态检查、单元测试或针对解析函数的最小脚本。
   - 如缺少真实数据库或远端权限，则只做代码级路径验证，并明确缺口。
   - verify: 记录执行命令、结果和无法验证的原因。

6. 形成可行方案
   - 提供低风险修复方案：例如扩大系统消息扫描条件、修正游标推进、补扫窗口、稳定排序、失败批次不推进完成态。
   - 提供历史数据补救方案：例如按时间范围回扫、按群补扫、服务端去重幂等重放。
   - 标注每个方案的改动文件、兼容性、风险和验证方式。
   - verify: 生成 `docs/邀请统计增量扫描系统消息遗漏排查记录-2026-05-23.md`。

7. 汇总本次本地变更
   - 汇总新增/修改的文档文件。
   - 如果修改代码，记录具体文件、原因、验证结果；如果未修改代码，也明确说明。
   - verify: 生成 `docs/邀请统计增量扫描系统消息遗漏变更记录-2026-05-23.md`。

## 需要集中确认的问题触发条件

以下情况出现时暂停执行，并集中向用户确认：

- 需要修改 Supabase 表结构、索引、唯一键或 RLS/授权策略。
- 需要改变客户端与服务端同步协议或公共数据结构。
- 需要进行历史数据清理、删除、覆盖或大范围补写。
- 需要依赖真实微信 WCDB 数据库样本、线上 Supabase 数据或远端密钥。
- 发现“被邀请/退出系统消息”的业务口径存在多种解释，代码和文档无法自洽。
