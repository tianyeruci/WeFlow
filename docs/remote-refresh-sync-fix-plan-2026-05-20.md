# 轻量远程刷新、本地同步修复与群人数分页执行计划

## Summary
- Web 按钮名称固定为“刷新最新数据”。
- 本地端每 5 秒轮询 WeFlow-Web，看是否有远程刷新请求。
- 有主动刷新能力后，本地自动同步改为：服务启动时立即同步一次，之后每 5 分钟同步一次。
- Web 页面 dashboard 每 10 秒自动刷新一次，从 Supabase 拉取最新展示数据。
- 手动同步和远程触发同步使用当前账号邀请统计数据全量 upsert；定时同步仍走增量。
- 不改表，复用 `sync_batches` 存刷新请求、处理状态和同步日志。

## Key Changes
1. Web 数据大屏新增 `刷新最新数据` 按钮，点击后按 IP 做 15 秒冷却，允许多人请求进入刷新列表。
2. `WeFlow-main` 新增 5 秒远程刷新轮询，通过现有 `REMOTE_SYNC_TOKEN` 领取最新请求，只处理最新一条并完成回写。
3. 本地自动同步调整为启动立即执行一次增量同步，之后每 5 分钟执行一次。
4. 手动同步和远程触发同步改为全量 upsert；同步前先刷新有效/无效、raw events、群人数、群名等本地派生数据。
5. dirty 增量同步补强：脏群绑定会携带对应活动标签，Web 同步接口写入 `sync_batches` 日志。
6. 刷新请求采用懒清理：创建或领取请求时清理昨日 `web-refresh:` 请求。
7. `WeFlow-main` 和 `WeFlow-Web` 的群人数展示增加分页，每页 10 条，按群人数从高到低。

## Test Plan
- Web 首次点击 `刷新最新数据` 能创建请求，15 秒内再次点击提示剩余冷却时间。
- Web 创建请求后，本地 5 秒内能领取，多条请求只处理最新一条，完成后状态写回 `completed` 或 `failed`。
- 服务启动时立即触发一次增量同步，后续间隔为 5 分钟。
- 手动同步和远程触发同步发送全量统计 payload，群人数、群名、有效/无效、退群状态能更新到远端。
- 两端群人数展示每页 10 条，排序从高到低，切换活动标签后回到第 1 页。
- 验证命令：`WeFlow-main npm run typecheck`，`WeFlow-Web npm run typecheck && npm run build`。

## Assumptions
- 不改表结构。
- Web 触发刷新不做登录，只按 IP 冷却。
- 本地端必须保持运行，并配置好现有远程同步地址和 token。
- Web 仍然只从 Supabase 读取；`刷新最新数据` 只是通知本地端把最新本地统计同步上去。
- 全量推送是当前账号邀请统计数据的全量 upsert，不是数据库全量覆盖，也不会删除远端历史孤儿数据。
