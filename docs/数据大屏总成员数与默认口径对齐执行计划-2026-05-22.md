# 数据大屏总成员数与默认口径对齐执行计划

## Summary

- 基于当前已有改动继续，不回退上一轮邀请排行榜与刷新链路修复。
- 目标：WeFlow-main 默认口径与 WeFlow-Web 一致；WeFlow-main 只额外保留“去重”可选开关。
- 完成后保存变更记录到 `docs/数据大屏总成员数与默认口径对齐变更记录-2026-05-22.md`。

## Key Changes

- 总成员数：
  - Web 与 Main 默认都按当前活动标签下绑定群的 `member_count` 合计。
  - Main 在 `dedupeMembers=false` 时不再用溯源邀请事件数推导总成员数。
  - Main 的 `dedupeMembers=true` 分支保留现有去重逻辑，作为额外可选能力。

- 总成员数（含退出）：
  - Web 与 Main 默认都按公式计算：`活动标签下群成员总人数 + 溯源数据中状态为退出状态的人数 + 今日退出人数`。
  - “溯源数据中状态为退出状态”按成员溯源同一状态判断，即 `quit` 状态，包括退出事件和已标记退群的邀请溯源行。
  - “今日退出人数”沿用北京时间当天 00:00:00 起的今日退群统计。
  - Main 只有在开启去重时继续走原有去重口径，不改去重开关的历史含义。

- 邀请人数排行榜默认口径：
  - Main 服务层和导出默认改为不去重，和 Web 默认一致。
  - Main 页面现有去重开关保留；开启后才使用去重排行榜。
  - 邀请排行榜的非去重逻辑继续沿用上一轮修复：按邀请溯源行计数，不看状态和归因，排除 `direct`，未知邀请人归入“未知来源”。

## Implementation Notes

- Web：调整 `getDashboard` 中 `totalMembersWithQuit` 的计算，不再使用 `scopedInviteEvents.length`。
- Main：调整 `getDashboard` 默认 `dedupeMembers` 解析为仅 `true` 时启用去重，并让默认含退群总数使用新公式。
- Main：调整 `exportInviteRanking` 默认值，未传 `dedupeMembers` 时按不去重导出。
- 抽出或复用退出状态统计 helper，避免总成员数（含退出）与成员溯源状态判断分叉。

- 根目录：`git diff --check`
- 手工核对：
  - Web 与 Main 默认未勾选去重时，总成员数一致。
  - 总成员数 = 当前活动标签下所有绑定群 `member_count` 合计。
  - 勾选“含退群”后 = 总成员数 + 溯源 quit 状态人数 + 今日退出人数。
  - Main 勾选去重后仍按原去重逻辑表现。
  - Main/Web 默认邀请排行榜人数一致，并与上一轮非去重邀请溯源口径一致。

## Assumptions

- “总成员数（含退出）”按用户给出的字面公式执行，即今日退出人数即使也出现在退出状态溯源中，也按公式额外相加。
- 这次不新增数据库表、不改 Supabase schema。
- 不调整刷新链路、事件源兼容、多微信合并等上一轮已完成逻辑。
