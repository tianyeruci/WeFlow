# 导出后 WCDB 连锁打开失败排查与修复执行计划（2026-05-23）

## 背景

当前链路表现为：邀请统计页正常 -> 创建导出任务后出现 `WCDB 打开失败` -> 聊天模块异常 -> 再进入邀请统计页也报 `WCDB 打开失败`。

初步判断问题不在邀请统计解析本身，而更像是导出 worker、聊天、邀请统计、群分析等路径在主进程里共用 WCDB 原生资源时，连接生命周期和重连时序没有统一管理。

## 当前判断

- 导出 worker 启动前，主进程里的 WCDB 连接没有被完整释放。
- `chatService` 里存在把“worker 存在”误判成“数据库已连接”的逻辑。
- `chatService.close()` 目前不是异步等待式收尾，容易出现关闭未完成就重新打开的竞态。
- `imageDecryptService` 也有同类连接状态误判。
- 导出期间，主进程里仍可能出现新的 WCDB 打开请求，和导出 worker 的生命周期冲突。

## 执行步骤

1. 落实本地排查计划与变更记录
   - 将当前执行计划保存到 `docs/导出聊天邀请统计WCDB打开失败执行计划-2026-05-23.md`
   - 后续完成后再保存变更记录到 `docs/导出聊天邀请统计WCDB打开失败变更记录-2026-05-23.md`
   - verify: 本地文档可打开，内容完整。

2. 增加主进程 WCDB 访问门禁
   - 新增内部模块 `WeFlow-main/electron/services/wcdbAccessGate.ts`
   - 在主进程导出期间，对 WCDB 打开请求做排队等待
   - 由导出入口持有独占门禁，避免新的主进程打开请求抢占时序
   - verify: 导出期间主进程 WCDB 打开请求会等待，导出结束后恢复。

3. 在导出 worker 启动前释放主进程 WCDB
   - 在 `WeFlow-main/electron/main.ts` 的导出入口里，先等待邀请统计后台任务空闲
   - 关闭聊天消息 cursor
   - `await wcdbService.close()`
   - 导出完成或失败后释放门禁
   - verify: 导出前主进程连接已关闭，导出后可重新打开。

4. 修复连接状态误判与关闭竞态
   - `WeFlow-main/electron/services/chatService.ts`
   - `WeFlow-main/electron/services/imageDecryptService.ts`
   - `chatService` 不再用 `wcdbService.isReady()` 代表“已连接”
   - 改为 `await wcdbService.isConnected()`
   - `chatService.close()` 改为异步等待式关闭
   - verify: 关闭后不会误判为已连接，重新进入聊天/邀请统计时可正常重连。

5. 补强错误信息记录
   - 导出失败、聊天连接失败、邀请统计成员加载失败时保留 `getLastInitError()` 的细节
   - 在本地变更记录中写清楚根因、触发路径、修复点和验证结果
   - verify: 错误日志能定位到真实失败原因，而不是只剩通用的 `WCDB 打开失败`。

6. 运行静态验证
   - `npm run typecheck`
   - `git diff --check`
   - verify: 两项命令均通过。

7. 整理本次变更记录
   - 将实际修改过的文件、核心修复点、验证结果写入 `docs/导出聊天邀请统计WCDB打开失败变更记录-2026-05-23.md`
   - verify: 变更记录可直接用于回溯。
