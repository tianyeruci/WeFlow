# WeFlow 邀请统计与成员溯源需求梳理

## 1. 背景

需要在 WeFlow 中新增一套“邀请统计 / 成员溯源”能力，用于读取本地微信数据中指定微信群的系统消息，识别成员入群、邀请入群、扫码入群、退群等事件，并生成类似参考图中的数据大屏、发售群列表、群成员溯源页面。

参考页面：

- `http://101.126.132.177:5173/#/dashboard`
- `http://101.126.132.177:5173/#/trace`

WeFlow 已具备本地微信数据读取、消息解析、群成员读取、联系人展示名和头像读取等基础能力。本需求应优先复用 WeFlow 现有能力，在 WeFlow 内新增页面和本地统计逻辑。

## 2. 核心结论

- 实现方式：直接在 WeFlow 内新增功能页面和服务逻辑。
- 数据来源：第一版以微信本地系统消息解析结果为准。
- 活动模型：使用“活动标签”管理微信群，同一个微信群只能归属一个活动标签。
- 展示筛选：展示页面必须选择一个具体活动标签，不提供跨活动聚合选项。
- 人员统计：同一活动标签下可能包含多个群，同一个人可能同时存在于多个群；除群成员溯源外，所有人数类统计必须在活动标签范围内使用有效数据，按 `activity_tag_id + wx_id` 去重，避免重复计数。
- 时间筛选：展示页面按时间范围筛选统计结果。
- 扫描方式：不提供扫描时间范围选择。增量扫描默认按当前活动标签和 `group_id`，分别从原始记录表中该群最后一条 `invite_time` 往后扫描；若该群没有历史记录，则扫描该群全部可读取系统消息。全局扫描用于当前活动标签全量重扫，扫描前先按活动标签删除原始记录，再重新读取该活动标签下多个群的全部可读取系统消息。
- 退群口径：单独维护退群事件表，因为同一成员可能加入多个群，只退出其中一个群。
- wxid 识别：优先使用消息结构和群成员数据中的 wxid；无法直接获得时用昵称、群昵称、备注、历史绑定匹配；无法唯一确认时进入待确认记录。
- 飞书定位：现有飞书表作为历史数据参考和后续同步目标，第一版不强制自动写入飞书。

## 3. 产品目标

- 支持按活动标签查看群增长、邀请排行、退群情况和实时动态。
- 支持配置活动标签，并把微信群分配到某一个活动标签下。
- 支持按时间范围查看统计结果。
- 支持查询某个成员由谁邀请、何时进入哪个群，以及该成员后续又邀请了多少人。
- 支持处理无法自动识别 wxid 的待确认记录。
- 支持导出邀请排行榜、群列表、成员溯源和原始事件数据。

## 5. 活动标签规则

### 5.1 活动标签定义

活动标签用于把微信群划分到不同活动中，例如：

- 拉新活动。
- 定金活动。
- 训练营活动。
- 发售活动。

系统不提供跨活动聚合选项。运营需要先选择一个具体活动标签，再查看该活动标签覆盖群内的统计结果。

背景口径：初期分销进群人数较多，筛选合格人员后会重新进入新的群并参加其它活动，后续活动的人数通常会越来越少。因此同一个 `wx_id` 可以出现在不同活动标签中，但统计时只在当前活动标签范围内处理，不做跨活动合并。

### 5.2 互斥规则

- 同一个微信群只能打一个活动标签。
- 如果某个群已经归属 `拉新活动`，则不能再归属 `定金活动`。
- 修改群的活动标签时，需要从原活动标签中移出，再加入新活动标签。
- 未打标签的群不参与统计，除非用户先给该群选择活动标签。

### 5.3 展示筛选规则

展示页面只允许选择一个活动标签：

- 选择某个活动标签：只统计该标签下的微信群。
- 不提供跨活动标签聚合或多活动标签同时选择。

展示页面不再提供“群集合”多选或二级集合筛选，避免同一群重复统计。

### 5.4 人员去重规则

同一个活动标签中存在多个群，同一个人员可能在多个群中出现。大屏和排行榜中的人数统计不能按群成员记录简单相加，必须先在当前活动标签范围内完成去重。

去重优先级：

1. 优先使用 `wx_id` 作为人员唯一标识。
2. 如果 `wx_id` 缺失，但存在人工确认绑定，使用绑定后的 `wx_id`。
3. 如果没有 `wx_id`，使用 `activity_tag_id + normalized_display_name` 临时聚合，但该结果需要标记为可能存在误差。
4. 同名且无法确认身份的记录进入待确认，不参与确认口径的人数统计，或单独计入待确认数量。

示例：

```text
同一个 wx_id 同时在拉新活动的 16 群、37 群、备用群中：
- 总成员数仅在群口径下计 1 人
- 勾选含退群后，总成员数含退群口径下计 1 人
- 今日新增计 1 人
- 邀请排行榜中作为被邀请人只贡献 1 次
```

### 5.5 有效入群记录规则

原始入群事件必须完整保留。除群成员溯源和原始事件明细外，所有统计均使用“有效入群记录”。

有效入群记录的生成范围固定为当前选择的活动标签：

1. 先根据 `activity_tag_id` 找到该活动标签下的所有群。
2. 从这些群的 `invite_events` 中取状态为 `confirmed` 且 `valid_flag = 1` 的入群记录。
3. 按 `activity_tag_id + wx_id` 分组。
4. 同一组中取 `invite_time` 最小的一条作为有效入群记录。
5. 如果 `invite_time` 相同，使用 `source_create_time`、`source_message_id` 或 `id` 做稳定排序兜底。

有效记录决定统计归因：

- 数据大屏、进群时段分布、今日新增、总成员数含退群口径、邀请排行榜均基于有效入群记录统计。
- 如果最早有效记录没有邀请者，例如普通入群，则该成员计入成员增长，但不计入邀请排行榜。
- 同一成员在同一活动标签下后续加入其它群的记录保留在群成员溯源中，但不重复参与人数统计。
- 待确认记录在人工确认前不进入有效入群记录；确认后重新参与有效记录计算。

### 5.6 退群后的原始记录状态

`invite_events` 需要同时维护两个状态字段：

- `delete_flag`：表示这条原始入群记录对应的单群成员关系是否已退出。`1` 为已删除 / 已退出，`-1` 为未删除 / 仍在该群。
- `valid_flag`：表示这条原始入群记录是否参与当前活动标签的人数统计和邀请归因。`1` 为有效，`-1` 为无效。

退群事件写入 `quit_events` 后，需要按 `activity_tag_id + wx_id` 回查原始入群记录，并按查询结果分支处理：

1. 如果同一 `activity_tag_id + wx_id` 只查到 1 条原始入群记录，说明该成员只存在于该活动标签下的一个群。根据这条记录的唯一 `id` 精确更新：`delete_flag` 改为 `1`，`valid_flag` 改为 `-1`，该记录对应邀请者的有效邀请人数减少 1。
2. 如果同一 `activity_tag_id + wx_id` 查到多条原始入群记录，继续在这些记录中查找 `group_id` 等于本次退群群 ID 的那一条。根据该条记录的唯一 `id` 更新：`delete_flag` 改为 `1`。
3. 多条记录场景下，如果更新后该成员在同一活动标签下仍存在其它 `delete_flag = -1` 的记录，说明成员只是退出了其中一个群，仍在该活动标签的其它群中；`valid_flag` 和邀请归因保持不变，邀请排行榜不减人数，也不重新归因到其它群的邀请者。
4. 多条记录场景下，如果更新后已不存在其它 `delete_flag = -1` 的记录，说明成员已从该活动标签下的所有群退出；将该成员当前有效入群记录的 `valid_flag` 改为 `-1`，该有效记录对应邀请者的有效邀请人数减少 1。

因此：

- `delete_flag` 是单群维度状态，用于判断某条入群明细是否已退该群。
- `valid_flag` 是活动维度统计状态，用于判断该成员是否仍参与该活动标签的人数统计和邀请归因。
- 群成员溯源仍展示完整原始明细，不因为 `delete_flag` 或 `valid_flag` 删除记录。
- 退群更新必须使用原始记录唯一 `id` 定位具体行，不能只按昵称或 wx_id 批量更新。

## 6. 页面需求

### 6.1 数据大屏

数据大屏用于查看当前选择活动标签下的整体邀请与成员增长情况。

页面顶部不展示“星轨 / 星轨系统”品牌区，也不展示“本地微信已连接”和“设置”。顶部仅保留页面导航，并将 `D 数据大屏`、`G 发售群列表`、`T 群成员溯源`、`P 待确认记录` 作为清晰的大标题导航。

#### 顶部筛选

- 活动标签：单选，仅包含用户创建且已启用的活动标签。
- 统计时间：使用一个时间范围组件选择开始时间和结束时间，用于筛选统计结果。
- 增量扫描按钮：按当前活动标签和 `group_id` 的增量规则扫描当前活动标签覆盖的群。
- 全局扫描按钮：用于当前活动标签全量重扫；扫描前按活动标签删除原始记录，再扫描该活动标签下所有群的全部可读取系统消息。

不需要展示扫描时间范围。
顶部筛选区不展示“总成员含退群”复选框，也不展示全局导出按钮；总成员数口径切换放在总成员数指标卡内完成。

#### 指标卡片

以下指标必须随当前选择的活动标签变化：

- 监控群组：当前活动标签下微信群数量。
- 总成员数：合并原“当前在群成员”和“含退群成员”两项。默认展示仅在群人数；在总成员数指标卡内勾选“显示含退群人数”后展示含退群人数。两种口径都按 `wx_id` 去重。
- 今日新增：当前活动标签下当天首次出现有效入群记录的人员数量，按 `wx_id` 去重。
- 今日退群：当前活动标签下当天发生退群的人员数量，按 `wx_id` 去重。
- 待确认记录：当前活动标签下待确认事件数量。

#### 入群时段分布

展示时间范围内的有效入群数量分布。默认按小时聚合。

统计受以下条件影响：

- 当前活动标签。
- 时间范围。

#### 邀请人数排行榜

展示当前活动标签和时间范围内的邀请人排行。排行榜只使用有效入群记录；同一 `activity_tag_id + wx_id` 的后续入群记录不再重复贡献邀请人数。

字段：

- 排名。
- 邀请人昵称。
- 邀请人 wxid。
- 邀请人数。
- 关联群数量。
- 最近邀请时间。

交互：

- 支持柱状图。
- 支持饼图。
- 支持最小邀请人数过滤，例如只显示邀请人数大于等于 N 的邀请人。
- 支持导出。

#### 群人数展示

展示当前活动标签下群成员数排行。

字段：

- 群名。
- 群 ID。
- 当前成员数。
- 今日新增。
- 今日退群。
- 活动标签。

#### 实时动态 / 最近事件

展示最近解析到的入群或退群事件。

字段：

- 事件类型：邀请 / 扫码 / 直接入群 / 退群 / 被移出。
- 用户昵称。
- 邀请者昵称。
- 群名。
- 时间。
- 解析状态：已确认 / 待确认。

### 6.2 发售群列表

发售群列表用于维护微信群的活动标签。

#### 列表字段

- 群名。
- 群 ID。
- 当前成员数。
- 今日新增。
- 今日退群。
- 活动标签。
- 最近扫描时间。
- 最近入库邀请时间。

#### 操作

- 为未归属群设置活动标签。
- 修改已归属群的活动标签。
- 清除活动标签，使该群不参与统计。
- 搜索群名。
- 按成员数排序。
- 按今日新增排序。
- 导出群列表。

### 6.3 群成员溯源

群成员溯源用于查询成员来源。

群成员溯源表格必须展示完整明细记录，不做人员去重，也不做 `group_id + wx_id` 合并。同一成员加入多个群、同一成员在同一群多次入群或退群，都需要按实际记录逐条展示。

#### 筛选条件

- 活动标签：单选。
- 群：在当前活动标签下选择具体某一个群。
- 成员昵称。
- 成员 wxid。
- 时间范围。
- 精确匹配。
- 是否包含已退群。

#### 表格字段

- 成员昵称 `user`。
- 成员 wxid `wx_id`。
- 邀请者 `inviter`。
- 邀请者 wxid `inviter_wx_id`。
- 所在群 `group_name`。
- 群 ID `group_id`。
- 入群时间 `invite_time`。
- 成员头像 `head_img`。
- 入群类型 `join_type`。
- 该成员作为邀请者邀请的人数。
- 状态：来自 `delete_flag` 的业务展示，`-1` 显示“未退出”，`1` 显示“退出”，无法确认时显示“待确认”。
- 是否有效：来自 `valid_flag` 的业务展示，`1` 显示“有效”，`-1` 显示“无效”，无法确认时显示“待确认”。

表格中的每一行对应一条可追溯的入群或退群记录；允许同一 `wx_id`、同一昵称、同一群在不同时间出现多行。

#### 交互

- 点击成员查看该成员全部入群、退群完整记录。
- 点击邀请者查看邀请者贡献明细。
- 点击群名筛选该群。
- 支持分页。
- 支持导出。

### 6.4 待确认记录

待确认记录用于处理无法唯一识别 wxid 的邀请或退群事件。

进入待确认的情况：

- 昵称匹配不到任何群成员。
- 在被邀请者加入的目标群内，昵称匹配到多个群成员。
- 邀请者或被邀请者缺失。
- 系统消息格式无法解析完整。

待确认字段：

- 原始消息。
- 解析出的用户昵称。
- 解析出的邀请者昵称。
- 目标群内候选用户列表。
- 候选邀请者列表。
- 所在群。
- 活动标签。
- 消息时间。

操作：

- 选择正确成员。
- 标记为未知。
- 忽略该记录。
- 保存确认结果，后续同群同昵称优先复用。

## 7. 数据来源与解析对象

### 7.1 微信本地数据

第一版主要读取 WeFlow 已解密并可查询的本地微信消息数据，重点是指定微信群中的系统消息。

优先读取：

- `localType = 10000`：系统消息。
- 其它包含 `<sysmsg>` 或可读系统文本的消息类型，作为兼容补充。

### 7.2 需要识别的系统消息

第一版至少支持以下文本模式：

```text
"A"邀请"B"加入了群聊。
"A"邀请你和"B、C、D"加入了群聊。
"A"通过扫描"B"分享的二维码加入群聊。
"A"加入了群聊。
"A"退出了群聊。
"A"被"B"移出了群聊。
```

实际微信系统消息可能包含 XML、HTML entity、群成员 wxid、换行、特殊空格或多语言变体，解析前需要先做文本清洗。

### 7.3 飞书已有数据格式

用户现有飞书表中已有邀请原始数据，字段如下：

```text
id
user
inviter
inviter_wx_id
invite_time
group_name
group_id
wx_id
head_img
delete_flag
valid_flag
```

飞书链接：

```text
https://mcnjs068wecx.feishu.cn/base/LntlbpEiDaHC6HsMUEkc8bACn21?from=from_copylink
```

其中 `id` 是每条原始邀请记录的唯一 ID，用于退群后精确更新某一条原始记录；`delete_flag` 建议新增到原始数据表，`1` 表示已删除 / 已退出该群，`-1` 表示未删除 / 仍在该群；`valid_flag` 表示是否参与活动级有效统计，`1` 为有效，`-1` 为无效。

第一版不要求自动写入飞书，但本地字段设计应兼容上述字段，便于后续同步。

## 8. 数据模型

### 8.1 活动标签表 `invite_activity_tags`

| 字段 | 类型建议 | 说明 |
| --- | --- | --- |
| `tag_id` | string | 活动标签 ID |
| `tag_name` | string | 活动标签名称 |
| `enabled` | boolean | 是否启用 |

## 9. 解析规则

### 9.1 邀请入群

```text
"A"邀请"B"加入了群聊。
```

| 字段 | 值 |
| --- | --- |
| `user` | B |
| `inviter` | A |
| `join_type` | `invite` |

多人邀请也按同一邀请者拆成多条原始记录：

```text
"A"邀请你和"B、C、D"加入了群聊。
```

应生成 4 条 `invite_events`：

| user | inviter | join_type |
| --- | --- | --- |
| 你 | A | `invite` |
| B | A | `invite` |
| C | A | `invite` |
| D | A | `invite` |

拆分后的每条记录都保留同一条原始消息、同一 `source_message_id` 和各自的被邀请人字段；统计时仍按当前活动标签下的 `activity_tag_id + wx_id` 取最早有效记录，避免同一人跨群重复贡献。

## 10. wxid 匹配策略

1. 如果消息结构中能直接获得 wxid，直接使用。
2. 如果消息结构中没有 wxid，先只在被邀请者加入的目标群成员列表中，用群昵称或当前展示昵称精确匹配。
3. 如果目标群内只有一个同名候选，自动确认该候选的 `wx_id`。
4. 如果目标群内存在多个同名候选，且无法从消息结构、群成员缓存或历史绑定中进一步确认唯一 `wx_id`，进入待确认记录。
5. 如果目标群内没有候选，再使用历史群成员缓存和人工确认过的 `member_identity_bindings` 尝试补充匹配。
6. 邀请者 wxid 同样优先从消息结构和目标群成员中匹配；无法唯一确认时进入待确认或标记为邀请者待绑定。
7. 如果没有候选，进入待确认。

原则：

- 不为了补齐 wxid 强行误归因。
- 同名成员不是一律进入待确认；只有在入群所在群内存在多个同名候选且无法唯一确认时，才进入待确认。
- 人工绑定结果应可复用，但仅在相同群或明确相同 wxid 场景下复用。

## 11. 去重规则

### 11.1 邀请事件去重

事件入库去重优先使用消息级唯一键，避免重复扫描写入同一条原始事件。

优先去重键：

```text
group_id + source_message_id
```

如果没有微信消息 ID，则使用：

```text
group_id + source_local_id + source_create_time
```

兜底去重键：

```text
group_id + wx_id + inviter_wx_id + invite_time
```

当 wxid 缺失时：

```text
group_id + user + inviter + invite_time + raw_content
```

### 11.2 退群事件去重

优先去重键：

```text
group_id + wx_id + quit_time
```

当 wxid 缺失时：

```text
group_id + user + quit_time + raw_content
```

### 11.3 统计展示去重

入库事件需要保留逐条完整记录。数据大屏、排行榜等人数类汇总统计必须在当前活动标签范围内使用 `valid_flag = 1` 的有效入群记录；群成员溯源明细和原始事件明细不去重，必须逐条展示。

推荐统计唯一键：

```text
activity_tag_id + wx_id
```

同一 `activity_tag_id + wx_id` 存在多条入群记录时，只取 `invite_time` 最小的一条作为有效记录。后续同一成员进入该活动标签下其它群的记录仍保留在群成员溯源中，但不进入大屏和排行榜重复统计。

如果同一个人跨多个活动标签都出现，该成员会在各自活动标签下分别计算；不同活动标签之间不互相去重。

当 `wx_id` 缺失时，统计使用人工绑定结果；没有绑定的记录不进入确认人数口径，进入待确认记录。

## 12. 统计口径

### 12.1 活动标签过滤

所有统计先根据当前选择活动标签确定群范围：

```text
某活动标签 = tag_id 对应的群
```

再根据时间范围过滤 `invite_time` 或 `quit_time`。入群相关统计使用该活动标签范围内 `valid_flag = 1` 的有效入群记录；退群相关统计按该活动标签范围内的 `quit_events` 按 `wx_id` 去重。

### 12.2 今日新增

统计当天 `effective_invite_events` 中首次入群的去重人员数量。

默认包含：

- `invite`
- `qrcode`
- `direct`

同一人员当天进入同一活动标签下多个群时，只计 1 人，取该活动标签内最早一条有效入群记录作为今日新增归属记录。

### 12.3 今日退群

统计当天 `quit_events` 中发生退群的去重人员数量。

默认包含：

- `self_quit`
- `removed`
- `unknown`

同一人员当天退出同一活动标签下多个群时，大屏今日退群只计 1 人；群列表中的单群今日退群仍按该群自身记录展示。

退群同时会更新对应原始入群记录的状态：

- 先按 `activity_tag_id + wx_id` 查询该成员在当前活动标签下的原始入群记录。
- 如果只查到 1 条，按该条记录唯一 `id` 更新 `delete_flag = 1`、`valid_flag = -1`，并从对应邀请者的有效邀请人数中扣减 1。
- 如果查到多条，再按本次退群的 `group_id` 命中具体原始记录，只将该条记录的 `delete_flag` 改为 `1`。
- 多条记录场景下，只要该成员仍有其它 `delete_flag = -1` 的记录，就说明没有完全退出该活动标签，`valid_flag` 和邀请归因保持不变。
- 多条记录场景下，如果该成员已没有其它未删除入群记录，则将活动级有效记录的 `valid_flag` 改为 `-1`，并从对应邀请者的有效邀请人数中扣减 1。

### 12.4 总成员数

总成员数合并原“当前在群成员”和“含退群成员”两个指标，通过总成员数指标卡内的复选框切换口径。

默认不勾选“显示含退群人数”时，展示“总成员数仅在群”：

- 优先使用 WeFlow 当前群成员列表读取结果。
- 按当前活动标签下的群合并后，使用 `wx_id` 去重。

如要从事件表推导，则逻辑为：

```text
活动标签内 valid_flag = 1 且至少存在一条 delete_flag = -1 的去重人员
```

勾选“显示含退群人数”时，展示“总成员数含退群”：

- 统计当前活动标签下出现过入群事件的去重人员数。
- 不因退群而排除。
- 同一成员加入同一活动标签下多个群时，按 `wx_id` 只计 1 人；历史上曾有效参与过该活动标签的成员仍纳入含退群口径。

第一版建议默认展示 WeFlow 当前群成员数，事件表用于历史趋势和含退群口径。

### 12.5 邀请排行榜去重口径

按邀请者聚合：

```text
inviter_wx_id 优先，inviter 兜底
```

统计 `join_type` 为以下值的记录：

- `invite`
- `qrcode` 且存在明确二维码分享者

不默认把 `direct` 计入邀请排行榜。

邀请人数排行榜中的“邀请人数”统计有效入群记录中的被邀请人员去重数。同一被邀请人在同一活动标签下加入多个群，只按最早有效记录给对应邀请者计 1 人。

如果同一被邀请人在同一活动标签下出现多个邀请者归因，按最早一条已确认入群记录归因；其它记录保留在溯源明细中，但不重复计入排行榜。

### 12.6 扫码入群归因

对于：

```text
"A"通过扫描"B"分享的二维码加入群聊。
```

归因为：

```text
user = A
inviter = B
join_type = qrcode
```

对于无法识别分享者的扫码入群，归为：

```text
inviter = 未知来源
join_type = qrcode
```

### 12.7 成员溯源

成员溯源应支持同一成员加入多个群、退出其中一个群的情况。

成员溯源明细按事件记录展示，用于查看该成员每一次入群、退群及邀请来源变化。该明细不按 `wx_id`、昵称、`group_id + wx_id` 去重；但页面顶部人数统计、趋势图和排行榜仍使用活动标签内的有效入群记录。

## 13. 扫描机制

### 13.1 增量扫描

页面不提供扫描时间范围选择。

点击“增量扫描”后：

1. 根据当前选择活动标签确定需要扫描的群。
2. 对当前活动标签下的每个群，按 `activity_tag_id + group_id` 查询原始记录表中该群最大的 `invite_time`。
3. 如果该群存在历史入库记录，则以该群最大的 `invite_time` 作为扫描起点，继续读取之后的系统消息。
4. 如果该群没有历史入库记录，则扫描该群全部可读取系统消息。
5. 同一 `invite_time` 附近可能存在多条系统消息，实际游标需结合 `source_message_id`、`source_create_time` 或本地唯一 `id` 做稳定去重，避免重复写入或漏扫。
6. 解析出的邀请事件和退群事件写入本地表。
7. 扫描结束后更新 `last_scan_time`、`last_invite_time`、`last_message_id`。

该规则以“当前活动标签 + 单个群”为游标维度，能避免某个群已有新记录后导致其它群的历史系统消息被跳过。

### 13.2 全局扫描

页面提供“全局扫描”按钮，用于当前活动标签的全量重扫。

点击“全局扫描”后：

1. 根据当前选择活动标签确定该活动标签下的全部群。
2. 扫描前按 `activity_tag_id` 从原始记录表删除该活动标签下已有入群原始记录。
3. 同步删除或重置该活动标签下由这些原始记录派生的待确认记录、有效统计缓存和扫描游标，避免旧结果残留。
4. 重新读取该活动标签下所有群的全部可读取系统消息。
5. 按解析规则生成新的原始入群记录、退群记录和待确认记录。
6. 扫描结束后重新计算该活动标签下的 `delete_flag`、`valid_flag`、有效入群记录和页面统计。

全局扫描只影响当前活动标签，不删除其它活动标签的数据；因为一个群只能归属一个活动标签，所以不会跨活动误删。

### 13.3 扫描后的自动处理

增量扫描或全局扫描完成后，后续数据处理应自动执行，不应要求用户再手动触发其它处理动作。

自动处理流程：

1. 解析系统消息，生成原始入群记录、退群记录和待确认记录。
2. 对原始事件做消息级去重，避免重复扫描产生重复记录。
3. 对可确认成员补齐 `wx_id`、`inviter_wx_id`、头像和群信息。
4. 按 `activity_tag_id + wx_id` 计算有效入群记录：同一活动标签下多条记录只保留 `invite_time` 最小的一条为 `valid_flag = 1`，其余统计无效但仍保留在群成员溯源。
5. 根据退群事件回写原始记录的 `delete_flag`，并在成员完全退出该活动标签下所有群时，将有效记录的 `valid_flag` 置为 `-1`。
6. 重新计算当前活动标签下的大屏指标、趋势图、邀请排行榜和发售群列表。
7. 刷新页面展示，并更新扫描日志。

### 13.4 触发方式建议

第一版建议同时保留“增量扫描”和“全局扫描”按钮。增量扫描作为用户主动补扫和排查问题的入口；全局扫描用于当前活动标签数据异常、解析规则升级或首次需要重建历史数据时使用。

同时建议增加本地后台自动扫描机制：

- WeFlow 启动并检测到本地微信已连接后，自动对已配置活动标签的群执行一次轻量增量扫描。
- WeFlow 运行期间，每 3 分钟执行一次后台静默增量扫描。
- 后台扫描只使用上述增量规则，不自动执行全局扫描，也不提供扫描时间范围选择。
- 同一时间只允许一个扫描任务运行，避免重复写入和状态竞争。
- 自动扫描失败时记录日志并在页面展示最近失败原因，不阻断用户手动点击“增量扫描”。

这样用户不需要一直点击按钮，数据会持续靠近实时；按钮主要用于立即同步、首次初始化、或发现数据异常时手动补扫。

### 13.5 手动刷新

手动刷新只重新计算当前活动标签和时间范围下的统计结果，不改变扫描起点。

### 13.6 实时监听

第二阶段可接入 WeFlow 现有消息监控能力，在新系统消息到达时自动解析。

第一版不强制实时监听，避免引入过多状态复杂度。

## 16. 异常与边界情况

### 16.1 同名成员

同名成员需要按“入群所在群”缩小匹配范围。目标群内只有一个同名成员时，可自动确认该成员的 `wx_id`；目标群内存在多个同名成员，且消息结构、历史群成员缓存或人工绑定都无法确认唯一身份时，才进入待确认。

### 16.2 改群昵称

历史系统消息中的昵称可能与当前群昵称不同。需要保留原始昵称，并允许人工绑定。

### 16.3 多群入群

同一成员可加入多个群。溯源明细按事件逐条展示；活动标签统计按 `activity_tag_id + wx_id` 取最早有效记录，不重复计算该成员在同一活动标签下后续加入的其它群。

### 16.4 只退一个群

同一成员退出其中一个群时，只影响该群的状态，不影响该成员在其它群的状态。

### 16.5 删除或缺失历史消息

如果本地微信历史系统消息不完整，则无法还原完整历史邀请链路。页面应展示最近扫描时间和最近入库邀请时间，避免误解为全量事实。

### 16.6 无法获取 wxid

允许记录只有昵称，但状态为 `pending` 或 `confirmed_without_wxid`。统计时需要明确标识该类记录可能存在合并误差。

## 17. 验收标准

第一版完成后，应满足以下标准：

1. 用户可以创建、启用、停用活动标签。
2. 用户可以为微信群设置一个且仅一个活动标签。
3. 已归属某活动标签的群不能同时归属另一个活动标签。
4. 展示页面只能选择某一个具体活动标签，不提供跨活动聚合选项。
5. 一个活动标签可以包含多个群，但一个群只能出现在一个活动标签下。
6. 展示页面可以通过一个统计时间范围组件筛选开始时间和结束时间。
7. 监控群组、总成员数、今日新增、今日退群、待确认记录会随活动标签变化。
8. 同一活动标签下同一人员存在于多个群时，除群成员溯源外，所有统计都只使用最早一条有效入群记录。
9. 总成员数默认展示仅在群人数；在总成员数指标卡内勾选“显示含退群人数”后展示含退群人数，复选框状态需要与该指标卡文案和数值联动。
10. 数据大屏不展示独立的“有效邀请人”指标卡。
11. 页面不要求用户选择扫描时间范围。
12. 点击增量扫描时，按当前活动标签和 `group_id` 查询各群最后一条 `invite_time` 后继续扫描；某个群没有历史记录时，扫描该群全部可读取系统消息。
13. 系统可以识别邀请入群消息。
14. 系统可以识别扫码入群消息，并将二维码分享者作为邀请者。
15. 系统可以识别普通入群消息。
16. 系统可以识别退群和被移出群消息。
17. 系统可以生成本地邀请事件表。
18. 本地邀请事件表包含唯一 `id`、`delete_flag` 和 `valid_flag`；退群后能先按 `activity_tag_id + wx_id` 查询，再按单条或多条记录场景更新单群删除状态和活动级有效状态。
19. 当同一活动标签下该成员只有 1 条原始入群记录时，退群会将该记录 `delete_flag` 改为 `1`、`valid_flag` 改为 `-1`，并扣减该记录邀请者的有效邀请人数。
20. 当同一活动标签下该成员有多条原始入群记录时，退群先按 `group_id` 命中具体记录并只更新该条 `delete_flag = 1`；若成员仍在其它群中，邀请者数据不扣减。
21. 系统可以生成本地退群事件表。
22. 数据大屏可以展示核心指标、入群时段分布、邀请人排行、群人数排行和最近动态。
23. 发售群列表可以展示微信群、活动标签和扫描状态。
24. 群成员溯源可以查询成员来源、邀请者、入群时间和所在群，明细记录完整展示且不按人员或群维度去重。
25. 被邀请者 wxid 缺失时，系统会先在被邀请者加入的目标群内按昵称匹配；目标群内唯一命中可自动确认，目标群内多候选或无法确认 wxid 时才进入待确认列表。
26. 用户可以人工确认待确认记录。
27. 用户可以导出邀请排行榜、成员溯源和原始事件数据。
28. 点击全局扫描时，系统会先按当前活动标签删除原始记录，再重扫该活动标签下所有群的全部可读取系统消息。
29. Windows 和 macOS 均沿用 WeFlow 现有本地微信读取能力运行。

## 18. 推荐实施阶段

### 阶段一：活动标签与群配置

- 新增活动标签表。
- 新增群活动标签绑定表。
- 实现同一群只能绑定一个活动标签。
- 实现活动标签单选逻辑，移除跨活动聚合入口。

验证方式：

- 能创建活动标签。
- 能给群设置活动标签。
- 同一群不能被重复分配到多个活动标签。

### 阶段二：系统消息增量扫描与解析

- 根据活动标签确定群范围。
- 按当前活动标签和 `group_id` 读取各群最后一条 `invite_time` 进行增量扫描。
- 实现当前活动标签的全局扫描：先删除该活动标签原始记录，再重扫标签下所有群。
- 实现邀请入群解析。
- 实现扫码入群解析。
- 实现普通入群解析。
- 实现退群解析。
- 实现事件入库去重和有效入群记录计算。

验证方式：

- 使用样例消息可生成正确结构化事件。
- 重复扫描不会生成重复记录。
- 有历史数据时只扫描当前活动标签下对应群最后一条 `invite_time` 之后的新增系统消息。
- 全局扫描只删除并重建当前活动标签的数据，不影响其它活动标签。

### 阶段三：wxid 匹配与待确认

- 优先使用消息结构中的 wxid，缺失时基于被邀请者加入的目标群成员匹配 wxid。
- 实现目标群内同名冲突识别。
- 实现待确认列表。
- 实现人工绑定。

验证方式：

- 目标群内单一匹配可自动确认。
- 目标群内多候选进入待确认。
- 人工确认后同类记录可复用绑定。

### 阶段四：数据大屏

- 活动标签单选筛选。
- 时间范围筛选。
- 指标卡片。
- 入群时段分布。
- 邀请人排行榜。
- 群人数排行。
- 最近动态。

验证方式：

- 切换活动标签后，指标和图表随群范围变化。
- 修改时间范围后，统计结果随时间变化。

### 阶段五：发售群列表与成员溯源

- 发售群列表。
- 活动标签维护。
- 成员溯源查询。
- 分页和搜索。

验证方式：

- 能按成员昵称 / wxid 查询。
- 能展示邀请者、群、入群时间、成员状态。

### 阶段六：导出

- 邀请排行榜导出。
- 成员溯源导出。
- 原始事件导出。

验证方式：

- 导出文件字段完整。
- 中文内容正常显示。

### 阶段七：飞书同步

- 接入飞书 API。
- 同步邀请事件。
- 同步退群事件。
- 记录同步状态。
- 支持失败重试。

验证方式：

- 本地记录能写入飞书。
- 重复同步不会重复创建。
- 同步失败有错误信息。

## 19. 待确认问题

正式开发前建议确认：

1. 第一版是否完全不写飞书，仅预留字段。
2. 是否需要从飞书导入历史邀请记录，与本地解析结果合并。
3. “直接入群”是否计入今日新增，但不计入邀请人排行榜。
4. 无法识别二维码分享者时，是否统一归为“未知来源”。
5. 待确认记录是否需要批量确认能力。
6. 是否需要支持多个微信账号分别统计。
7. 数据保存位置采用 WeFlow 现有配置存储，还是新增本地 SQLite。
