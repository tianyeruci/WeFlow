# 邀请人数排行榜展示与全部活动口径对齐计划（2026-05-21）

## Summary
- 不改表结构，不改同步链路，只修 `WeFlow-main` 的排行榜展示和 `WeFlow-Web` 的全部活动聚合口径。
- 目标是让主端排行榜每个名字都完整出现，同时让 Web 在选择“全部活动”时按和主端一致的绑定关系汇总，避免少算。

## Key Changes
- `WeFlow-main/src/pages/InviteStatsPage.tsx`
  - 让邀请人数排行榜的柱状图高度随排行条数自适应，强制每条排行都渲染对应名字，不再靠 ECharts 自动跳过标签。
  - 保持现有柱状图 / 饼图切换、配色和去重开关不变。
- `WeFlow-Web/lib/supabase-rest.ts` + `WeFlow-Web/lib/invite-data.ts`
  - 增加分页拉全的 Supabase 读取方式，用于 `final_stat_events` / `group_tag_bindings` 的全量汇总，避免单次查询上限截断。
  - “全部活动”继续作为 synthetic scope，但聚合时按主端同样的有效绑定关系收口：按 `group_id + activity_tag_id` 过滤有效事件，group binding 按 `group_id` 去重并保留最新 `updated_at`。
  - 保持 Web 只读、默认非去重展示和现有标签默认选择规则不变。
- 文档
  - 执行阶段把计划和变更记录写入本地 `docs/`，便于回溯。

## Test Plan
- `WeFlow-main` 跑 `npm run typecheck` 和 `npm run build`。
- `WeFlow-Web` 跑 `npm run typecheck` 和 `npm run build`。
- 手工核对同一账号下的 `全部活动`、单活动、排行榜前三项、总成员数、入群人数与主端一致。
- 在 `WeFlow-main` 的排行榜里确认每一条 inviter 名字都可见，不再出现隔行才显示的情况。

## Assumptions
- 远端 Supabase 已经是当前可见账号范围内的数据源；Web 只修聚合和展示，不补本地未同步数据。
- `全部活动` 不改成真实标签，仍是合并所有活动标签下群数据的语义。
- 执行开始先保存本计划到本地，执行完成后另存变更记录。
