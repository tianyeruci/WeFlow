# 数据大屏默认标签与查询性能修复执行计划

## Summary

- 基于当前已有改动继续，不回退邀请排行榜、刷新链路、总成员数口径修复。
- 目标：
  - 页面端默认进入“拉新”活动标签，并且首屏请求的数据就限定在拉新标签下。
  - Dashboard 与群成员溯源使用同一批最新兼容事件数据，避免排行榜人数、入群人数与溯源条数不一致。
  - 降低页面端查询量，减少“远程数据暂不可用”、502、浏览器页面卡死。
- 完成后保存变更记录到 `docs/数据大屏默认标签与查询性能修复变更记录-2026-05-22.md`。

## Key Changes

- 默认标签：
  - Web 初始 `selectedTagId` 不再默认为 `__all__`，等待活动标签加载完成后再选择默认标签。
  - 默认标签优先选择名称为“拉新”的活动标签；没有“拉新”时选择第一个可用活动标签；没有标签时才回退到“全部活动”。
  - 标签下拉中把“拉新”排在第一项，避免 UI 显示与首屏数据作用域错位。
  - `selectedTagId` 未就绪时不请求 Dashboard、群列表轮询或成员溯源。

- Web 查询性能：
  - 事件读取优先使用 `invite_events` / `quit_events` 兼容表；仅当兼容表没有数据时才 fallback 到 `final_stat_events`。
  - 成员溯源按 `tagId + groupId` 下推到远端查询，选中群时不再先拉取整个活动标签的数据。
  - 成员溯源接口增加分页参数，页面默认分页拉取，降低单次响应体积和浏览器渲染压力。
  - 成员溯源的群筛选选项改从 `group_tag_bindings` 构建，避免按群过滤后下拉选项只剩当前群。

- 最新数据与统计一致性：
  - Dashboard、邀请排行榜、成员溯源继续共用同一套兼容事件标准化逻辑。
  - 排行榜非去重口径继续沿用“邀请溯源行”逻辑：排除 `direct`，不看状态/归因，未知邀请人归入“未知来源”。
  - 成员溯源分页只影响页面列表读取，不影响导出接口的完整导出。

## Test Plan

- `WeFlow-Web`: `npm run typecheck`
- `WeFlow-main`: `npm run typecheck`
- 根目录：`git diff --check`
- 手工核对：
  - 首次打开页面时，活动标签默认选中“拉新”，首个 Dashboard 请求携带拉新标签 id，而不是 `__all__`。
  - 拉新标签下的群人数、邀请排行榜、实时动态、成员溯源均只展示拉新标签绑定群的数据。
  - 选中单个群查看成员溯源时，响应速度明显快于全标签拉取，列表可分页浏览。
  - Dashboard 邀请排行榜入群人数与成员溯源中同标签、同群、同时间条件下的邀请溯源条数一致。
  - 页面刷新和切换筛选后不再出现长时间卡死或 502。

## Assumptions

- “拉新”活动标签名称以 `拉新` 为准；如果不存在，使用第一个可用活动标签作为默认。
- 这次不新增数据库表、不改 Supabase schema。
- 502 主要由首屏全量查询和成员溯源全量拉取导致，本轮先通过减少查询范围和分页解决。
