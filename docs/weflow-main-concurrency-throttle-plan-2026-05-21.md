# WeFlow-main 定时任务并发与节流重构计划（2026-05-21）

## 目标
- 仅调整 `WeFlow-main` 的定时调度、任务并发控制和页面反馈。
- 不改表结构，不改现有扫描范围和同步 payload 结构。
- 让「增量扫描」「检查是否退出群」「同步本地数据」都变成：
  - 到点触发
  - 同类型忙时静默跳过
  - 不排队、不补跑

## 已确认约束
- 增量扫描与退群检查分开两把锁。
- 同类型任务正在运行时，调用方只做静默跳过，不弹 busy toast。
- 自动调度、手动按钮、远端触发都走同一套并发控制。

## 实施步骤
1. 拆分任务锁
   - 增量扫描使用独立运行锁。
   - 退群检查使用独立运行锁。
   - 同步本地数据保留独立同步锁。
   - 运行态查询同时返回分开的运行状态。
2. 调整定时频率
   - 增量扫描：启动后第 3 分钟首次执行，之后每 30 秒一次。
   - 退群检查：启动后第 1 小时首次执行，之后每 1 小时一次。
   - 同步本地数据：启动后第 5 分钟首次执行，之后每 30 秒一次。
3. 收敛忙时语义
   - 同类型任务运行中时直接返回 skipped。
   - 页面端收到 skipped 后不弹提示。
   - 远端触发遇到忙时不排队，直接结束当前请求。
4. 补前端状态展示
   - 页面端分别判断增量扫描和退群检查的运行状态。
   - 按钮只跟随各自任务的忙闲状态。
   - 全局刷新与详情刷新在任务结束后照常进行。
5. 记录变更
   - 修改完成后补一份变更记录到 `docs/`。

## 验证方式
- `WeFlow-main` 执行 `npm run typecheck`
- `WeFlow-main` 执行 `npm run build`
- 手动确认：
  - 增量扫描与退群检查可独立运行
  - 同类型重复触发只静默跳过
  - 同步本地数据不再排队

## 风险点
- 任务并发后会同时改动同一份本地数据对象，需要保持运行态字段隔离。
- 忙时静默跳过后，远端请求需要明确完成或失败，避免挂起。
