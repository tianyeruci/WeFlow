# 邀请统计增量扫描系统消息遗漏排查记录（2026-05-23）

## 结论摘要

当前最可能导致“增量扫描完成后，被邀请或退出系统消息遗漏”的主因是：扫描水位使用了 `rawEvents.updated_at`，但 `updated_at` 不是稳定的消息时间，而是事件记录的更新时间。代码还会在“本轮没有新增 rawEvent”时，把某条历史 rawEvent 的 `updated_at` 推进到本轮扫描开始时间。这样会让下一轮用一个偏新的业务更新时间去过滤微信消息的 `message.createTime`，从而跳过仍未入库、但 `createTime` 早于该水位的系统消息。

第二个明确风险是：当前 WCDB 查询只读取 `localType = 10000`，虽然后续过滤函数写了 `senderUsername === group_id` 的兼容判断，但因为非 10000 消息根本没有被查出来，所以这条兼容逻辑实际覆盖不到非 10000 的系统消息。

如果用户看到的“遗漏”发生在 Web 页面或服务端视图，还需要确认遗漏层级：是 `raw_events` 没有记录，还是记录存在但因为 `status = pending/ignored`、`valid_flag`、业务视图过滤而没有展示。

## 证据链

### 1. 强证据：`updated_at` 水位会跳过未入库历史系统消息

相关代码：

- `inviteStatsService.ts` 中 `getGroupLastRawUpdatedAt(data, groupId)`：按该群所有 `rawEvents.updated_at` 取最大值。
- `scanActivityInternal(tagId)` 中：
  - `scanStart = this.getGroupLastRawUpdatedAt(data, binding.group_id)`
  - `if (scanStart > 0 && createTime <= scanStart) continue`
  - `if (scanStart > 0 && oldestMessageTime <= scanStart) reachedWatermark = true`
- 同函数中：
  - 如果本轮没有新增 rawEvent 且该群已有 rawEvent，执行 `advanceGroupRawUpdatedAt(data, group_id, startedAt)`。

这意味着扫描下界来自 rawEvent 更新时间，而不是来自“上次已经扫描到哪条微信消息”。

可复现场景：

1. 某群已有一条 rawEvent，`updated_at = 2026-05-23 10:00:00`。
2. 之后发现一条真实微信系统消息，`message.createTime = 2026-05-23 09:59:00`，但它之前没有成功解析或没有进入 rawEvents。
3. 下一轮增量扫描时 `scanStart = 10:00:00`。
4. 代码判断 `createTime <= scanStart`，直接跳过。
5. 该消息不会再被解析，也不会同步到远端。

这与 2026-05-22 变更记录中的剩余风险一致：`updated_at` 作为扫描水位后，若微信数据库中出现创建时间早于水位的晚到历史消息，会被跳过。

### 2. 强证据：空窗口会把扫描水位推进到当前时间

相关代码：

- `scanActivityInternal(tagId)`：
  - `let groupAddedRawEvent = false`
  - 新增 rawEvent 时才置为 true。
  - 扫描结束后，如果 `!groupAddedRawEvent && hasGroupRawEvents`，调用 `advanceGroupRawUpdatedAt(data, group_id, startedAt)`。
- `advanceGroupRawUpdatedAt()`：
  - 找该群 `updated_at` 最大的 rawEvent。
  - 如果它小于本轮 `startedAt`，调用 `markSyncDirty(latest, updatedAt)`。
  - `markSyncDirty()` 会把 `row.updated_at = now/updatedAt`。

影响：

- 即使本轮没有新增任何邀请/退出系统消息，也会把该群水位推进到扫描开始时间。
- 后续只要消息 `createTime <= startedAt`，都会被认为已经扫描过。
- 如果某些消息是“晚到本地数据库”、“之前解析规则不支持”、“之前 localType 没被查到”、“用户后来补了成员绑定才可确认”，这些消息仍可能被永久跳过。

### 3. 强证据：事件更新会污染扫描水位

`rawEvents.updated_at` 不只在扫描游标推进时更新，也会在以下场景更新：

- rawEvent 内容变化时，`upsertRawEventFromInvite()` / `upsertRawEventFromQuit()` 会 `markSyncDirty(existing, now)`。
- `recomputeFlags()` 会重算有效标记，并通过 `syncRawEventsFromMaterialized()` 同步 rawEvents。
- 人工确认、身份绑定、退群状态回写等流程也可能让相关事件变 dirty。

所以某条很老的 rawEvent 只要今天被人工确认或重算，它的 `updated_at` 就可能变成今天。下一轮扫描再用这个 `updated_at` 作为全群 `message.createTime` 水位，会跳过这条时间之前所有尚未入库的系统消息。

### 4. 强证据：非 10000 类型的系统消息当前查不到

当前扫描查询：

```text
wcdbService.getMessagesByType(group_id, 10000, false, 500, offset)
```

后续过滤：

```text
message.localType === 10000 || message.senderUsername === group_id
```

问题是查询阶段已经限定 `localType = 10000`，所以 `senderUsername === group_id` 无法让非 10000 类型的系统消息进入候选集。

如果微信数据库里某些“被邀请/退出”系统消息实际不是 `localType = 10000`，而是其它类型但内容含 `<sysmsg>`、`<plain>`、`<text>` 或可读系统文本，当前扫描一定漏掉。

### 5. 中证据：解析规则偏精确，变体消息会静默丢弃

当前 `parseSystemEvents()` 只识别固定格式：

- `A邀请B加入了群聊`
- `A邀请你加入了群聊，群聊参与人还有：B、C`
- `A通过扫描B分享的二维码加入了群聊`
- `A被B移出了群聊`
- `A退出了群聊`
- `A加入了群聊`

解析失败时直接 `continue`，不会保存原始消息，也不会进入待确认。

可能漏掉的变体包括：

- 系统消息有额外前后缀、括号、换行或非标准标点。
- 微信版本不同，文案不是“加入了群聊/退出了群聊/移出了群聊”。
- XML 中的可读字段不在 `<plain>` 或 `<text>`。
- “群聊参与人还有”中的成员是否应拆成独立邀请事件存在业务口径差异。

需要注意：2026-05-19 的记录明确写过，`A邀请你加入了群聊，群聊参与人还有：B、C、D` 这类背景消息当前只生成一条 `A -> 你` 的待确认邀请记录，其他成员只保留在上下文中。这是已有设计，不一定是代码 bug，但如果当前预期是把上下文成员也统计为被邀请人，需要调整业务口径。

### 6. 中证据：文档与代码对同步前扫描的描述不一致

2026-05-22 记录写到“所有同步入口在真正同步前都会等待或执行一次后台增量扫描”。

当前代码中：

- Web 刷新链路会在认领请求后调用 `ensureBackgroundScanComplete(10000)`。
- 普通 `queueSync()` 没有执行或等待扫描，只调用 `syncCurrentScope()`。

这不会解释“增量扫描已经完成但仍遗漏”的核心问题，但会导致某些手动同步/定时同步把尚未扫描的新数据同步到远端，表现为远端数据暂时旧。

### 7. 弱证据：offset 分页在消息表变化时可能不稳定

当前分页是倒序 + offset：

```text
limit 500, offset += result.rows.length
```

如果扫描期间微信消息表继续写入，offset 结果集可能发生移动。代码会跳过 `createTime > startedAt` 的新消息，但这些新消息仍可能改变后续 offset 的位置。这个风险相对低于水位问题，但属于增量扫描常见隐患。

### 8. 需区分：数据存在但页面/视图不展示

远端 Web 查询主要读取 `final_stat_events`。部分统计会过滤：

- `status = pending`
- `status = ignored`
- `valid_flag != 1`
- `delete_flag = 1`

因此如果 rawEvent 已同步，但成员 wxid 未确认或状态是 pending，用户可能在统计口径中感觉“漏了”。这类不是扫描漏查，而是展示/统计口径过滤。

## 可行方案

### 方案 A：把扫描游标与 rawEvent 更新时间彻底拆开（推荐）

核心调整：

- 不再用 `rawEvents.updated_at` 作为 `message.createTime` 下界。
- 在群绑定或本地扫描状态中维护专门的扫描游标，例如：
  - `last_scanned_message_time`
  - `last_scanned_local_id`
  - `last_scanned_server_id`
- 每轮按“消息时间 + 本地 ID/服务端 ID”推进游标。
- rawEvent 的 `updated_at` 只表示事件记录更新时间，不参与扫描边界。

优点：

- 从根上消除“业务更新时间污染扫描水位”的问题。
- 人工确认、有效性重算、同步状态变化不会影响扫描范围。
- 逻辑边界清晰，后续排查成本低。

风险：

- 需要修改本地数据结构。
- 如果要把新游标同步到远端顶层字段，可能涉及 Supabase 表结构；如果只保存在 `groupTagBindings.raw_json`，可减少远端结构改动。

验证：

- 构造旧 rawEvent `updated_at` 大于某条未入库系统消息 `createTime` 的样例，确认修复后仍能扫到。
- 构造人工确认更新 old rawEvent 的样例，确认扫描游标不变化。

### 方案 B：保留现有结构，但增加重叠回扫窗口（低风险止血）

核心调整：

- 当前 `scanStart` 不直接作为硬下界。
- 实际扫描下界使用：

```text
effectiveScanStart = max(0, scanStart - overlapSeconds)
```

- `overlapSeconds` 可先固定为 24 小时或 7 天。
- 依靠现有 dedupKey 避免重复写入。

优点：

- 改动小。
- 能覆盖一部分晚到消息、解析规则更新后的历史消息。

风险：

- 不能彻底解决很早历史消息被跳过的问题。
- 窗口越大，扫描越慢。
- 仍然保留 `updated_at` 混用水位的根因。

验证：

- 在水位前 1 小时放入未入库系统消息，确认能被补扫。
- 在水位前超过窗口的消息仍会被跳过，需要文档明确。

### 方案 C：取消空窗口推进 rawEvent.updated_at（建议与 A 或 B 同时做）

核心调整：

- 删除或停用 `advanceGroupRawUpdatedAt()` 对 rawEvent 的推进。
- 空窗口只更新 `binding.last_scan_time` 或专门 scan cursor。

优点：

- 避免“没有新增事件也把水位推到当前时间”。
- 改动点明确。

风险：

- 如果不引入专门游标，可能重新出现重复扫描空窗口的问题。
- 建议不要单独实施，最好与方案 A 或 B 配合。

### 方案 D：扩大候选消息查询范围

核心调整：

- 除 `localType = 10000` 外，增加对其它可能承载系统消息的候选查询。
- 理想做法是新增 native WCDB 查询能力：按 `sessionId + createTime range` 拉取候选消息，不只按 localType。
- 低风险替代做法是额外查询少量已知类型，并按内容包含 `<sysmsg>`、`<plain>`、`<text>`、`加入群聊`、`退出群聊`、`移出群聊` 做二次过滤。

优点：

- 覆盖非 10000 类型的系统消息。
- 让 `senderUsername === group_id` 的兼容逻辑真正生效。

风险：

- 全类型扫描可能明显变慢。
- native WCDB 接口改动需要更高验证成本。

### 方案 E：增加解析失败保留与样例测试

核心调整：

- 对 `localType=10000` 但解析失败的候选消息，至少在 scan log 或本地调试记录里保留计数/样本。
- 为 `parseSystemEvents()` 建立样例测试，覆盖：
  - 普通邀请
  - 多人邀请
  - 背景邀请
  - 扫码
  - 主动退出
  - 被移出
  - XML/HTML entity/换行/特殊标点

优点：

- 后续遇到遗漏能知道是“没查到”还是“查到但没解析”。
- 对业务文案变体更可控。

风险：

- 如果保存原始消息样本，需要注意隐私和本地文件大小。

### 方案 F：历史数据补救回扫

当前已经被水位跳过的消息，不会因为修复后自动回来，需要补救。

建议提供一个明确的“修复回扫”入口：

- 按活动标签 + 群 + 时间范围回扫。
- 忽略当前增量水位。
- 使用现有 dedupKey 幂等 upsert。
- 默认不删除旧数据，只补新增和更新。

可选回扫范围：

- 方案 1：从 2026-05-22 水位改造时间点前后开始回扫。
- 方案 2：按用户发现遗漏的具体群和日期范围回扫。
- 方案 3：对当前活动标签所有绑定群做一次全量候选回扫。

风险：

- 全量回扫耗时较长。
- 如果解析规则仍不覆盖某些文案，回扫也扫不出来。
- 如果业务要求修改“背景邀请参与人”的统计口径，需要先确认后再回扫，否则补出的数据不符合预期。

## 推荐落地顺序

1. 先确认遗漏层级：`raw_events` 是否缺失，还是只在页面/统计口径中缺失。
2. 先做方案 C + B 作为止血：取消空窗口推进 rawEvent 更新时间，并增加重叠回扫窗口。
3. 同步设计方案 A：新增专门扫描游标，彻底替换 `rawEvents.updated_at` 水位。
4. 如果样本证明存在非 10000 系统消息，再做方案 D。
5. 加解析样例测试和失败样本统计。
6. 提供一次按群/时间范围的历史回扫能力，修复已跳过的数据。

## 需要集中确认的问题

1. 当前“遗漏”是在 Supabase `raw_events` 表中没有记录，还是 `raw_events` 有记录但 Web 页面/统计没有展示？
2. 是否有 1 到 3 条真实遗漏系统消息样本，包括群 ID、消息时间、原始系统文本、是否能在本地微信聊天记录看到？
3. `A邀请你加入了群聊，群聊参与人还有：B、C` 这类背景邀请中，`B、C` 是否也要作为独立被邀请人进入统计？当前历史设计是不拆。
4. 历史补救更倾向按指定群/时间范围回扫，还是对当前活动标签做一次全量回扫？

## 当前判断

在没有真实数据样本前，代码层面已经能解释“符合条件但遗漏”的主要路径：`rawEvents.updated_at` 水位会跳过 `message.createTime` 更早的未入库消息。建议优先修正水位模型，再做历史补扫；否则即使临时补数据，后续仍可能因同一水位策略再次遗漏。

## 用户已确认

- `Supabase raw_events` 中确实没有这些遗漏消息。
- 背景邀请的参与人不拆，仍按人工拆分口径处理。
- 历史补救采用当前活动标签全量回扫，且会先清除原始数据再重新增量扫描。

## 对你提出的修法的判断

- 直接把 `updated_at` 改成邀请时间或退出时间，只能作为短期表象修正，不能作为最终方案。
- 原因是 `updated_at` 还会被人工确认、重算、同步 dirty、空窗口推进等流程改写；只要它还承担这些职责，就会再次污染扫描水位。
- 更稳的做法是把“扫描游标”和“业务更新时间”分开。
- 如果想保守落地，最低风险顺序是：
  1. 停止用 `rawEvents.updated_at` 直接过滤下一轮消息。
  2. 增加独立的群级扫描游标。
  3. 空窗口时只推进游标，不推进业务更新时间。
  4. 历史回扫时按活动标签全量清空后重扫。

## 性能补充

50 个群的增量扫慢，最可能的重耗时点不是消息分页本身，而是每个群都会先拉一次群成员并做联系人/昵称解析。

最稳的优化顺序：

1. 先把群成员加载改成“有候选系统消息才加载”。
   - 这不会改变没有新消息的群的结果。
   - 能直接省掉大量空扫时的群成员查询。
2. 再考虑给群成员结果加短期缓存或复用现有面板缓存。
   - 这能减少重复扫描同一批群时的重复查询。
   - 但缓存过久会带来昵称/成员变更的轻微滞后，不建议一上来就做很长 TTL。
3. 不建议一开始就把 50 个群并行扫开。
   - 这会把 WCDB、联系人解析和本地 CPU 一起打满，副作用更难控。

如果你要我继续往下做，我建议下一步优先落“扫描游标拆分 + 群成员延后加载”这两个点，其他都先不碰。
