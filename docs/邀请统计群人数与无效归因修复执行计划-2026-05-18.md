# 邀请统计群人数与无效归因修复执行计划

## Summary

- 修复本地端和远程端：群人数展示改为活动标签绑定群的真实当前群人数。
- 修复“无效”语义：`无效` 只代表已确认入群记录不参与统计归因；`待确认` 不能显示或筛选为无效。
- 修复历史默认 `valid_flag = -1` 导致“只入群一次也无效”的问题：成员溯源读取、导出、同步前都要重算有效标记。

## Interface Changes

- 本地 `InviteGroupTagBinding` 增加 `member_count`，保存每个绑定群的真实当前成员数。
- 远程同步 `groupTagBindings` payload 增加 `member_count`，写入 Supabase 已存在的 `group_tag_bindings.member_count` 字段。
- 远程 `DashboardData.groupRanking[].count` 含义改为真实群人数，不再由入群事件推导。

## Implementation Changes

- 本地端：
  - `buildGroupRows` 的 `member_count` 使用 `GroupChatInfo.memberCount`。
  - 新增或复用刷新逻辑，在群列表、大屏、远程同步前刷新绑定群 `member_count` 并标记同步脏数据。
  - `getMemberTrace`、`listPending`、`exportMemberTrace` 调用前执行 `recomputeFlags`，修正历史默认 `valid_flag = -1`。
  - 溯源筛选中，`无效` 只匹配 `status === confirmed && event_type === invite && valid_flag === -1`。
  - 溯源展示和导出中，`pending` 优先显示“待确认”，不要因 `valid_flag = -1` 显示“无效”。
- 远程端：
  - `getDashboard` 同时读取 `group_tag_bindings`，用活动标签绑定群作为监控群和群筛选来源。
  - `groupRanking` 从 `group_tag_bindings.member_count` 构建并排序，绑定群即使暂无入群事件也展示。
  - 邀请排行榜、总成员数、今日新增继续使用有效入群记录，不受真实群人数改动影响。
  - 成员溯源保持读取 `final_stat_events`，但依赖本地同步后的最新 `valid_flag`。

## Test Plan

- 本地运行 `npm run typecheck`。
- 远程运行 `npm run typecheck`。
- 验证活动标签下“群人数展示”显示微信群真实成员数，例如“发售最靠谱的人”显示当前群总人数。
- 验证“家松”这类已确认、未退出、同活动只入群一次的记录显示“有效”。
- 验证“沐兮663”这类未匹配 wxid 的待确认记录不出现在“无效”筛选中。
- 验证 `"A"邀请你和"B、C、D、E、F"` 拆分后，不同 `wx_id` 不会因为同一原始消息互相打无效。
- 验证同一活动标签下同一 `wx_id` 多次入群，仅最早未退出记录有效；完全退群后有效记录变无效。

## Assumptions

- Supabase 当前表结构已包含 `group_tag_bindings.member_count`，不新增迁移文件。
- `valid_flag = -1` 的含义保持为“不参与统计/归因”，不是解析失败。
- 退群事件不参与有效/无效归因筛选，只作为成员状态展示。
