- 当前任务覆盖：邀请排行榜统计口径、Web 页面事件数据读取、页面端“刷新数据”链路防重复触发。
- 完成后保存变更记录到 `docs/数据大屏邀请排行榜与刷新链路修复变更记录-2026-05-22.md`。

## Key Changes

- 邀请排行榜口径：
  - 排行榜和群成员溯源共用同一批“邀请溯源行”。
  - 不去重：只要是邀请入群溯源行就计数，不看 `status`、`valid_flag`、`delete_flag`、归因状态；每行 `+1`。
  - 去重：仅统计未退出群、`status=confirmed`、`valid_flag=1`、有被邀请人 wxid 的数据，并在当前活动标签/筛选范围内按被邀请人 wxid 去重。
  - `direct` 直接入群不计入邀请排行榜。
  - 没有明确邀请人的邀请溯源行归入“未知来源”，避免排行榜总数继续少于溯源邀请数。
  - Main 与 Web 两端同步调整，保持导出和页面展示一致。

- Web 页面数据缺失：
  - Web 事件读取优先兼容 `invite_events` / `quit_events`，并标准化为现有 `FinalStatEvent` 结构；`final_stat_events` 作为兼容补充或旧环境兜底。
  - 继续使用 `group_tag_bindings` / 活动标签过滤，确保标签、群筛选不变。
  - `inviter_identity_mappings` 已存在但可为空，空表不影响排行榜。

- “刷新数据”链路：
  - 保留只有页面按钮 `POST /api/invite/sync-request` 才能创建刷新请求的设计。
  - 给 `web-refresh:%` 请求增加有效期，只处理短时间内新建的 `requested` 请求；过期的 `requested/processing` 请求标记为过期或失败，不再触发扫描。
  - Main 轮询到远端刷新请求时，只有成功领取到“新鲜请求”后才调用扫描和全量同步。
  - 防止同一来源在已有未完成请求时反复创建新请求，避免几十秒一次扫描。

## Interfaces

- 不新增数据库表。
- `sync_batches.status` 增加使用已有字符串字段的新状态值：`expired` 或 `superseded`，用于刷新请求生命周期管理。
- Web 数据读取层新增兼容转换：把 `invite_events` / `quit_events` 转为页面现有统计事件结构，兼容 `inviter_wxid`、`inviter_wx_id`、`raw_json.join_type` 等字段。

## Test Plan

- `WeFlow-main`: `npm run typecheck`
- `WeFlow-Web`: `npm run typecheck`
- 根目录：`git diff --check`
- 手工核对：
  - 同一标签/群/时间范围下，不去重排行榜总邀请数 = 群成员溯源中邀请入群溯源行数量。
  - 退出群、无效归因、待确认数据进入不去重排行榜，但不进入去重排行榜。
  - 去重排行榜只按被邀请人 wxid 去重，且只统计未退出群有效数据。
  - 不手动点击“刷新数据”时，`sync_batches` 不产生新的有效 `web-refresh` 请求，Main 不触发扫描。
  - 点击一次“刷新数据”只触发一次领取、扫描、同步、完成流程。
  - Web 页面能重新展示邀请排行榜、最近动态、群成员溯源等事件类数据。

## Assumptions

- `inviter_identity_mappings` 表已创建但可为空，当前不新增管理 UI。
- 多微信邀请人合并沿用前一版方案：按映射合并展示并按合计排序。
- 这次不改变总成员数、今日新增、今日退群等其他卡片口径，除非实现中发现它们直接依赖同一处错误事件源。
