# 增量扫描、全部活动口径与排行榜统计修复计划

## Summary
- 本次只修上次改动带来的问题：增量扫描变慢、同步与扫描联动过度、排行榜日期筛选和标题统计不准、Main/Web 大屏口径不一致。
- 明确保留“全部活动”概念：`__all__` 是虚拟范围，不是真实活动标签，不改表结构。

## Key Changes
- 增量扫描：
  - 保留 `rawEvents.updated_at` 作为水位。
  - 继续只扫描系统消息，不扫描普通聊天；优先用现有 `localType=10000`，`senderUsername === groupId` 只作为兼容判断。
  - 把当前一次性读取全部系统消息改为倒序分页读取系统消息，每页固定数量，到 `createTime <= lastUpdatedAt` 立即停止该群扫描。
  - 首次扫描无历史 rawEvent 的群仍扫描历史系统消息，但只扫系统消息。
  - 无新增 rawEvent 且该群已有 rawEvent 时，继续推进该群最大 `updated_at` 到本轮扫描时间。
- 同步与刷新联动：
  - 增量扫描、同步本地数据、检查是否退出群的手动触发和定时触发保持独立。
  - 定时同步和 Main 手动同步不再自动等待或触发增量扫描。
  - 只有 Web 页面点击“刷新数据”对应请求被 main 认领后，才联动增量扫描。
  - Web 刷新联动规则：已有扫描则最多等 10 秒；没有扫描则触发一次扫描并最多等 10 秒；到时无论扫描是否结束都继续同步本地数据。
- 全部活动口径：
  - Main/Web 的“全部活动”都按有效 `group_id + activity_tag_id` 绑定过滤事件。
  - 群下拉展示可按 `group_id` 去重，但事件统计不能只取最新绑定；同一群绑定多个活动时，全部活动要包含这些有效活动绑定下的事件。
  - Main 去重开启时，全部活动按成员键去重，重复成员取最早有效入群记录。
  - Main 默认不去重；Web 保持默认不去重且不新增去重开关。
- 排行榜日期筛选与标题：
  - Web 的 `datetime-local` 在浏览器端转成带时区的绝对时间再请求，避免 Vercel UTC 解析偏移；Main 保持本地秒级时间戳。
  - 排行榜筛选只影响排行榜数据和排行榜标题，不误改顶部总卡片。
  - `【标签】招募者 xx 名，入群人数 xx` 中：
    - `招募者 xx 名` = 当前活动/全部活动、群、时间筛选后的排行榜行数。
    - `入群人数 xx` = 当前排行榜筛选后的 `invite_count/count` 合计。
  - 下载图片标题同步使用同一口径。
- 数据大屏统计口径：
  - Main/Web 都按当前活动标签或全部活动有效绑定范围统计，不直接拼 `rawEvents`。
  - Main 不去重时保持现有文档口径：展示当前范围内邀请人邀请到的事件数。
  - Main 去重时按有效成员键统计；“含退群”只影响总成员数卡片，不影响排行榜标题入群人数。
  - 校验今日新增、今日退群、总成员数、群人数、实时动态与近期文档口径一致。

## Test Plan
- `WeFlow-main`: `npm run typecheck`
- `WeFlow-Web`: `npm run typecheck`
- `git diff --check`
- 手工验收：
  - 已有水位的群，增量扫描只分页读取最近系统消息，到水位即停止。
  - Main 手动同步和定时同步不触发增量扫描。
  - Web 刷新最多等待增量扫描 10 秒，然后继续同步。
  - Main/Web 在单活动和全部活动下，同群同时间段的排行榜行数、招募者数量、入群人数一致。
  - 同一群绑定多个活动时，全部活动不会漏掉非最新绑定下的事件。

## Assumptions
- 微信系统消息类型仍以 `localType=10000` 为主。
- `全部活动` 仍是合并所有有效活动标签下群数据的虚拟范围。
- 本次不重做解析器，不改数据库结构，只优化读取范围、联动边界和统计展示口径。
