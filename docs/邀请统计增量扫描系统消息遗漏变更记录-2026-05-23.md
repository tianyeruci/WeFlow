# 邀请统计增量扫描系统消息遗漏变更记录（2026-05-23）

## 本次完成

- 修复了服务端增量扫描把 `rawEvents.updated_at` 当作消息水位的问题。
- 新增了独立的群级消息扫描游标，写入 `groupTagBindings` 的本地扩展字段：
  - `last_scanned_message_time`
  - `last_scanned_local_id`
- 空窗口扫描不再推动 raw event 的 `updated_at`。
- 增量扫描时，invite/quit 事件的 `created_at` / `updated_at` 现在跟随系统消息时间，不再统一写成本次扫描开始时间。
- 扫描时先解析候选系统消息，只有确实命中候选消息时才加载群成员，减少 50 群空扫的耗时。
- 页面端“增量扫描”入口改为触发所有活动标签扫描。
- 新增“当前活动标签全量回扫”服务端入口，先清空该标签范围内 raw / invite / quit 数据和本地游标，再走增量扫描。

## 这次确认的结论

- `Supabase raw_events` 没有缺失消息，问题主要在本地扫描链路。
- 原始漏扫的主因是扫描水位和业务更新时间混用了。
- `localType = 10000` 的查询限制仍然保留，因此非 10000 载体的系统消息仍是已知风险点。

## 已验证

- `npm run typecheck` 通过。
- `git diff --check` 没有报错。

## 仍然保留的风险

- 如果微信库里存在不以 `localType = 10000` 承载的邀请/退出系统消息，当前查询层仍可能漏掉。
- 全量回扫会清理当前活动标签范围的数据，执行前需要确认标签范围是对的。
