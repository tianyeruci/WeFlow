# 邀请人数排行榜统计口径与多微信合并计划

## Summary
- 覆盖范围：`WeFlow-main` + `WeFlow-Web` + Supabase schema/API。
- 执行阶段第一步先保存本计划到 `docs/邀请人数排行榜统计口径与多微信合并执行计划-2026-05-22.md`，完成后保存变更记录到 `docs/邀请人数排行榜统计口径与多微信合并变更记录-2026-05-22.md`。

## Key Changes
- 排行榜计数口径：
  - 不去重：统计当前标签/群/时间范围内所有明确邀请人产生的邀请/扫码入群明细，不看退出状态、`valid_flag`、归因有效性。
  - 去重：只统计 `status=confirmed`、`valid_flag=1`、未退出群的数据，并按被邀请人 wxid 去重。
  - 排行榜与群成员溯源同源，确保不去重时邀请人数量能和溯源明细对上。
- 多微信合并：
  - 新增后台维护表 `inviter_identity_mappings`，把多个 inviter wxid 映射到同一个 `person_key/person_name`。
  - 排行榜按映射后的同一人合并成一行，邀请人数展示合计，并按合计排序。
  - 未配置映射的邀请人仍按 wxid 优先、昵称兜底分组。
- 总成员/今日指标：
  - Main 保留现有去重开关并修正口径：不去重按标签下群成员数合计，去重按 wxid 去重。
  - Web 不新增总成员数去重开关；默认按标签下群绑定的 `member_count` 合计展示。
  - 今日新增、今日退群统一从北京时间 0 点开始统计。

## Interfaces
- Supabase 新增表：`inviter_identity_mappings`
  - 字段：`id`、`account_scope`、`person_key`、`person_name`、`wxid`、`display_name`、`enabled`、`created_at`、`updated_at`、`raw_json`。
  - 约束/索引：同一 `account_scope + wxid` 只映射到一个启用人员；按 `account_scope + person_key`、`account_scope + wxid` 建索引。
- Web 读取：
  - `getDashboard` 同时读取 `final_stat_events`、`group_tag_bindings`、`inviter_identity_mappings`。
  - 兼容 `inviter_wx_id` 与 `inviter_wxid` 字段名。
- Main 读取：
  - 增加本地 `inviterIdentityMappings` 数据结构。
  - 远端同步成功后从 Web/Supabase 拉取映射并写入本地副本；无映射时按旧逻辑运行。

## Test Plan
- `WeFlow-main`: `npm run typecheck`
- `WeFlow-Web`: `npm run typecheck`
- `git diff --check`
- 手工核对：
  - 同一邀请人在群成员溯源里的明细数，与排行榜不去重人数一致。
  - 退出群/无效归因记录不进入去重排行榜，但进入不去重排行榜。
  - 两个 wxid 映射到同一 `person_key` 后，排行榜合并成一行并按合计排序。
  - 今日新增/今日退群以北京时间当天 00:00:00 为起点。

## Assumptions
- 第一版多微信映射只通过后台数据表维护，不新增管理 UI。
- `direct` 直接入群不计入邀请排行榜。
- 远端“恢复初始化/清空统计数据”不删除 `inviter_identity_mappings`，因为它是人工维护配置，不是扫描产生的数据。
