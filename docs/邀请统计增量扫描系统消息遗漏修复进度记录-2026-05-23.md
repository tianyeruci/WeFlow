## 已完成

1. 已确认漏扫根因：
   - `rawEvents.updated_at` 被当成扫描游标使用。
   - 空窗口扫描会推动原有 raw event 的 `updated_at`。
   - 结果是下一轮扫描可能直接跳过尚未入库的系统消息。

2. 已修复服务端扫描逻辑：
   - 新增独立的群级扫描游标。
   - 不再用 raw event 的更新时间当消息水位。
   - 空窗口不再推进 raw event 更新时间。
   - 新增事件时间跟随系统消息本身时间。

3. 已优化性能：
   - 先解析候选消息。
   - 只有命中候选事件时才加载群成员。

4. 已调整页面入口：
   - “增量扫描”现在触发所有活动标签扫描。

5. 已补全历史回扫入口：
   - 新增 `rescanActivity(tagId)`。
   - 会先清理当前活动标签范围内的本地 raw / invite / quit 数据和游标，再重新扫描。

6. 已更新本地文档：
   - 查询流程记录
   - 变更记录
   - 进度记录

7. 已验证：
   - `npm run typecheck` 通过。
   - `git diff --check` 通过。

## 当前剩余风险

- WCDB 查询仍限定 `localType = 10000`，非 10000 载体的系统消息仍可能漏掉。
- 全量回扫会清理当前活动标签范围的数据，执行前要确认标签范围无误。

## 下一步

- 如果后续还要继续缩小漏扫范围，可以再补 WCDB 查询层对非 10000 系统消息的覆盖。
- 目前代码层已经把最容易导致“越扫越漏”的水位污染问题拆开了。
