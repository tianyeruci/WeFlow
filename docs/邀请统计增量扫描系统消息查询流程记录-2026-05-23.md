# 邀请统计增量扫描系统消息查询流程记录（2026-05-23）

## 范围

本记录描述当前 `WeFlow-main` 中“本地增量扫描 -> 本地入库 -> 远端同步 -> Web 查询展示”的实际流程。

## 入口

- 页面端“增量扫描”按钮：
  - `WeFlow-main/src/pages/InviteStatsPage.tsx`
  - 现在触发 `ALL_ACTIVITY_TAG_ID`，即所有活动标签扫描。
- 主进程 IPC：
  - `WeFlow-main/electron/main.ts`
  - `inviteStats:scanActivity`
  - `inviteStats:rescanActivity`
- 扫描实现：
  - `WeFlow-main/electron/services/inviteStatsService.ts`

## 当前增量扫描流程

1. 读取当前账号 scope 的本地统计数据。
2. 找到启用的活动标签和其绑定群。
3. 对每个群计算本地扫描游标：
   - `last_scanned_message_time`
   - `last_scanned_local_id`
4. 设定重叠窗口：
   - 当前实现会回扫最近 24 小时的系统消息，避免边界和晚到消息丢失。
5. 通过 WCDB 查询群消息：
   - `getMessagesByType(groupId, 10000, false, 500, offset)`
6. 经过本地映射后，保留候选系统消息：
   - `isInviteSystemMessage(message, groupId)`
7. 按消息时间和 localId 过滤当前扫描窗口。
8. 按 `createTime asc, localId asc` 排序。
9. 先解析文本：
   - `normalizeSystemContent(message)`
   - `parseSystemEvents(text)`
10. 只有命中候选事件时，才加载群成员：
    - `getGroupContext(data, binding)`
11. 构建邀请/退群事件：
    - `buildInviteEvent(...)`
    - `buildQuitEvent(...)`
12. 写入本地 raw 表：
    - `upsertRawEventFromInvite(...)`
    - `upsertRawEventFromQuit(...)`
13. 写入派生表：
    - `inviteEvents`
    - `quitEvents`
14. 扫描完成后更新绑定状态：
    - `last_scan_time`
    - `last_invite_time`
    - `last_message_id`
    - `last_scanned_message_time`
    - `last_scanned_local_id`

## 关键判断

- `rawEvents.updated_at` 不再作为下一轮扫描水位。
- 空窗口扫描不再推进 raw event 的 `updated_at`。
- 新增事件的业务时间跟随系统消息时间，不再统一使用扫描开始时间。
- 如果当前群没有候选系统消息，群成员不会被加载，空扫成本更低。

## 当前手动回扫流程

`rescanActivity(tagId)` 的逻辑是：

1. 校验只允许单个活动标签。
2. 清理该标签绑定群的：
   - `rawEvents`
   - `inviteEvents`
   - `quitEvents`
   - 本地消息游标
3. 重新执行同一套增量扫描。

## 已知风险

- WCDB 当前查询仍限定 `localType = 10000`。
- 如果存在非 10000 载体的系统消息，当前查询层仍可能漏掉。

## 结论

这次漏扫的核心不是远端同步，而是本地扫描游标设计问题。现在已经把“业务更新时间”和“消息扫描游标”拆开，后续若还要补历史消息，优先使用全量回扫入口。
