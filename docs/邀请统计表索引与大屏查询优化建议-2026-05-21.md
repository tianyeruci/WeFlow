# WeFlow 邀请统计表索引与大屏查询优化建议

记录时间：2026-05-21

## 结论先行

1. 你现在的大屏瓶颈，首先不是“索引不够”，而是 `GET /api/invite/dashboard` 这条链路会把 `final_stat_events` 基本整批拉回前端，再在 JavaScript 里做过滤、排序和聚合。
2. 所以，最有效的优化不是给每个字段都补索引，而是把大屏统计改成数据库可直接返回的聚合结果，或者至少改成物化视图 / 汇总表。
3. 在现有结构下，最值得继续加索引的主要是 `group_tag_bindings`，其次是 `raw_events` 的少量组合索引，以及少量兼容表的查询索引。
4. `final_stat_events` 是视图，不能直接建普通索引；要提速只能优化底表、改成物化视图，或者做汇总表。

## 当前高频查询链路

目前远程大屏的读取路径大致是：

- `WeFlow-Web/app/api/invite/dashboard/route.ts`
- `WeFlow-Web/lib/invite-data.ts`

关键点：

- `loadFinalEvents()` 直接读取 `final_stat_events`
- `loadGroupBindings()` 直接读取 `group_tag_bindings`
- 前端再计算 `cards`、`hourlyDistribution`、`inviteRanking`、`groupRanking`、`recentActivities`

这意味着：

- 现在最贵的是整表传输和前端聚合，不是单条 `where` 条件查询
- `limit 10000` 不是性能优化，只是截断结果；数据继续增长后会有漏数风险

## 现有索引状态

当前 schema 已经有一批基础索引，下面这些不用重复加：

- `activity_tags`: `account_scope`、`enabled`、`sync_status`
- `group_tag_bindings`: `account_scope + group_id` 唯一约束、`activity_tag_id`、`group_id`、`enabled`、`sync_status`
- `raw_events`: `account_scope + dedup_key` 唯一约束、`account_scope + group_id + created_time/invite_time/exit_time`、`event_type`、`member_wxid`、`related_wxid`、`sync_status`
- `invite_events` / `quit_events`: `activity_tag_id`、`group_id`、`member_wxid`、`inviter_wxid` / `operator_wxid`、时间列、`sync_status`
- `scan_logs`、`sync_batches`、`member_identity_bindings` 也已有基础索引

结论：不是所有表都需要继续堆索引。

## 最值得继续加索引的表

| 表 | 建议 | 原因 | 优先级 |
| --- | --- | --- | --- |
| `group_tag_bindings` | 增加 `activity_tag_id + group_id` 组合索引，最好只覆盖 `enabled = true` | 大屏先靠它缩小群范围，`final_stat_events` 的 join 也会吃到它 | P0 |
| `raw_events` | 现有索引基本够用；只有把统计下推到数据库后，再考虑新的组合索引 | 现在大屏并没有直接按 `raw_events` 做数据库级聚合 | P1 |
| `invite_events` / `quit_events` | 如果继续承担导出、溯源、审计查询，可补 `tag + time`、`tag + inviter/operator` | 适合排行榜、成员溯源、人工复核 | P2 |
| `activity_tags` | 一般不需要新增 | 行数通常很少，现有索引已够 | P3 |
| `member_identity_bindings` / `scan_logs` / `sync_batches` | 低优先级 | 主要是管理页和审计页，不是大屏热路径 | P3 |

### 具体建议 SQL

```sql
create index if not exists idx_group_tag_bindings_tag_group_enabled
  on group_tag_bindings(activity_tag_id, group_id)
  where enabled = true;
```

如果你后面把群排行也下推到数据库，可以再补：

```sql
create index if not exists idx_group_tag_bindings_tag_member_count
  on group_tag_bindings(activity_tag_id, member_count desc)
  where enabled = true;
```

如果 `invite_events` / `quit_events` 继续保留给导出和溯源：

```sql
create index if not exists idx_invite_events_tag_time
  on invite_events(activity_tag_id, invite_time);

create index if not exists idx_invite_events_tag_inviter_time
  on invite_events(activity_tag_id, inviter_wxid, invite_time);

create index if not exists idx_quit_events_tag_time
  on quit_events(activity_tag_id, exit_time);

create index if not exists idx_quit_events_tag_operator_time
  on quit_events(activity_tag_id, operator_wxid, exit_time);
```

## 比索引更有效的优化

1. 把大屏统计改成物化视图或汇总表。
   - 群统计：`activity_tag_id + group_id`
   - 排行榜：`activity_tag_id + inviter_wxid`
   - 时段分布：`activity_tag_id + hour/day`
2. 让后端直接返回聚合结果，不要再 `select * from final_stat_events` 后前端自己算。
3. 只查需要的列，不要 `select *`。
4. 最近动态单独查，不要把全部事件拉回来后再排序。
   - 更好的做法是数据库侧 `order by event_time desc limit 20`
5. 给 dashboard 接口加短 TTL 缓存，避免高频刷新重复打 DB。
6. 如果数据量继续增长，再考虑按时间或账号作用域分区；这是最后一层，不是第一层。

## 如果你要把 `final_stat_events` 改成物化视图

这时才值得给“最终统计层”建更强的索引，通常是：

- `activity_tag_id`
- `(activity_tag_id, invite_time)`
- `(activity_tag_id, group_id)`
- `(activity_tag_id, inviter_wxid)`

这会比在底表上继续堆很多单列索引更直接。

## 落地顺序

1. 先补 `group_tag_bindings` 的组合索引。
2. 再把 dashboard 统计下推到数据库汇总层，至少把卡片、排行、时段分布和最近动态拆开。
3. 最后评估 `invite_events` / `quit_events` 的补充索引和短缓存。

## 结论

如果只选一个动作，我会先做 `group_tag_bindings(activity_tag_id, group_id)` 的组合索引。

如果允许做一次结构优化，我会优先把大屏统计改成汇总表或物化视图，因为这比继续给底表加索引更能稳住性能。
