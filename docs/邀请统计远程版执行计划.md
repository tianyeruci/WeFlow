# WeFlow 邀请统计远程版执行计划

## 1. 总览

本计划用于将当前本地邀请统计扩展为远程用户版和远程管理版。

最终架构：

- 本地 WeFlow 负责读取微信本地系统消息。
- 本地 WeFlow 取消全局扫描，只保留增量扫描。
- 本地 WeFlow 将扫描得到的原始消息和机器解析结果同步到远程 API。
- 远程数据库作为邀请统计唯一事实源。
- 远程应用部署在 Vercel 或类似平台。
- 普通用户只读。
- 管理员管理远程数据库中的统计数据。

核心原则：

- 原始系统消息只作为凭据，不直接参与统计。
- 统计展示只读取最终统计视图。
- 一条原始消息可以人工修复为多条统计明细。
- 管理员修复结果不能被本地后续同步覆盖。

## 2. 阶段一：本地扫描规则调整

### 2.1 移除全局扫描

需要修改本地现有邀请统计：

- 移除页面中的全局扫描按钮。
- 移除 `scanActivity(tagId, 'full')` 调用。
- 移除 IPC / preload / 类型中对外暴露的 `full` 扫描模式。
- 后台自动扫描继续使用增量扫描。
- 扫描日志中不再产生全局扫描类型。

### 2.2 优化增量扫描游标

当前扫描不能只依赖 `invite_time`。新的扫描游标必须同时覆盖邀请和退出。

规则：

```text
对每个 activity_tag_id + group_id：
  last_created_time = 原始消息表中该范围内最大的 created_time

如果 last_created_time 不存在：
  扫描该群全部可读取系统消息
否则：
  扫描 created_time > last_created_time 的系统消息
```

只保存包含邀请或退出群聊含义的系统消息。

### 2.3 本地原始事件字段

本地原始数据需要具备以下口径：

- `created_time`：微信系统消息时间，作为扫描游标。
- `invite_time`：邀请消息时间；非邀请为空。
- `exit_time`：退出消息时间；非退出为空。

邀请消息：

```text
event_type = invite
created_time = 系统消息时间
invite_time = 系统消息时间
exit_time = null
```

退出消息：

```text
event_type = exit
created_time = 系统消息时间
invite_time = null
exit_time = 系统消息时间
```

如果当前本地仍使用 `inviteEvents` 和 `quitEvents` 两个集合，可以先兼容实现：

- 邀请集合新增或映射 `created_time = source_create_time`。
- 退出集合新增或映射 `created_time = source_create_time`。
- 游标计算同时查询邀请集合和退出集合的最大 `source_create_time`。

## 3. 阶段二：远程数据库设计

远程数据库推荐使用 Supabase、Neon 或 Vercel Postgres。Vercel Serverless 函数不保存长期文件。

### 3.1 基础表

需要建立：

- `accounts`：微信账号或数据源范围。
- `activity_tags`：活动标签。
- `group_tag_bindings`：群与活动标签绑定。
- `raw_group_system_messages`：原始系统消息凭据。
- `stat_events`：统计明细。
- `event_overrides`：人工修正记录。
- `sync_batches`：同步批次。
- `scan_logs`：扫描日志。
- `audit_logs`：管理员操作审计。

### 3.2 原始消息表

`raw_group_system_messages` 只作为凭据，不直接参与统计。

关键字段：

- `raw_message_id`
- `account_scope`
- `activity_tag_id`
- `group_id`
- `group_name`
- `source_message_id`
- `source_local_id`
- `created_time`
- `raw_content`
- `parsed_content`
- `event_hint`
- `sync_batch_id`
- `created_at`
- `updated_at`

唯一键优先级：

- `account_scope + activity_tag_id + group_id + source_message_id`
- `account_scope + activity_tag_id + group_id + source_local_id + created_time`

### 3.3 统计明细表

`stat_events` 保存可参与统计的明细。一条原始消息可对应多条统计明细。

关键字段：

- `event_id`
- `raw_message_id`
- `source_type`：`machine` / `manual`
- `event_type`：`invite` / `exit`
- `activity_tag_id`
- `group_id`
- `user`
- `wx_id`
- `inviter`
- `inviter_wx_id`
- `operator`
- `operator_wx_id`
- `invite_time`
- `exit_time`
- `status`
- `valid_flag`
- `delete_flag`
- `created_at`
- `updated_at`

### 3.4 人工修正表

`event_overrides` 保存管理员解释原始消息的动作。

支持动作：

- `confirm`：确认机器解析结果。
- `split`：一条原始消息拆成多条统计明细。
- `ignore`：忽略原始消息。
- `replace`：替换机器解析结果。

人工修正优先级高于机器解析。本地同步不能覆盖人工修正。

### 3.5 最终统计视图

建立 `final_stat_events` 视图或等价查询层。

规则：

```text
如果 raw_message_id 存在人工修正：
  使用 source_type = manual 的统计明细
否则：
  使用 source_type = machine 且 status = confirmed 的统计明细

排除：
  status = pending
  status = ignored
```

所有展示、导出和统计都读取最终统计视图。

## 4. 阶段三：同步 API

### 4.1 本地上传接口

本地 WeFlow 主动调用远程 API。

建议接口：

- `POST /api/sync/invite-stats/batch`
- `POST /api/sync/invite-stats/scan-log`
- `GET /api/sync/invite-stats/status`

上传内容：

- 活动标签。
- 群绑定。
- 原始系统消息。
- 机器解析统计明细。
- 扫描日志。

### 4.2 同步写入规则

- 活动标签按远程 ID 或稳定业务键 upsert。
- 群绑定按 `account_scope + group_id` upsert。
- 原始消息按唯一键 upsert。
- 机器解析统计明细可 upsert。
- 人工修正统计明细不可被同步覆盖。
- 同步批次需要记录成功、失败、耗时和错误信息。

### 4.3 本地首次迁移

启用远程模式时：

1. 本地读取现有邀请统计 JSON。
2. 生成一次全量同步批次。
3. 上传活动标签、群绑定、原始消息和机器解析结果。
4. 远程生成最终统计视图。
5. 本地后续管理改为读写远程 API。

## 5. 阶段四：远程用户版

### 5.1 页面

只实现：

- 数据大屏。
- 群成员溯源。

不实现：

- 发售群列表。
- 待确认记录。
- 标签管理。
- 扫描操作。
- 修复操作。

### 5.2 权限

普通用户 token 只允许只读接口。

禁止：

- 写活动标签。
- 写群绑定。
- 写统计明细。
- 写人工修正。
- 创建扫描任务。
- 处理待确认。

## 6. 阶段五：远程管理版

### 6.1 页面

实现完整页面：

- 数据大屏。
- 发售群列表。
- 群成员溯源。
- 待确认记录。

### 6.2 待确认处理

待确认记录以原始消息为单位展示。

管理员可以：

- 查看原始消息凭据。
- 确认机器解析结果。
- 将一条原始消息拆成多条统计明细。
- 替换机器解析结果。
- 忽略原始消息。

保存后：

- 写入 `event_overrides`。
- 创建或更新 `source_type = manual` 的 `stat_events`。
- 最终统计视图立即使用人工结果。

### 6.3 审计

所有管理操作写入 `audit_logs`。

需要记录：

- 操作人。
- 操作时间。
- 操作对象。
- 操作前摘要。
- 操作后摘要。
- IP 或客户端标识。

## 7. 阶段六：远程扫描任务增强

第一版不要求远程直接触发扫描。

后续可增加：

- `scan_jobs` 表。
- 远程管理员创建增量扫描任务。
- 本地 WeFlow 轮询任务。
- 本地 WeFlow 执行扫描并同步结果。
- 远程页面展示任务状态。

由于全局扫描已取消，远程扫描任务只支持增量扫描。

## 8. 本地管理端远程化

本地邀请统计页面进入远程模式后：

- 数据读取远程 API。
- 管理操作写远程 API。
- 扫描仍由本地执行。
- 扫描结果同步到远程数据库。
- 断网时可缓存扫描原始消息，但不允许管理修改。

## 9. 测试计划

### 9.1 本地扫描

- 本地页面不再展示全局扫描。
- 增量扫描会扫描活动标签下所有绑定群。
- 无历史原始消息的群会扫描全部可读取系统消息。
- 有历史原始消息的群只扫描 `created_time` 更大的系统消息。
- 只有退出记录、没有邀请记录的群不会重复从头扫描。
- 邀请消息写 `invite_time`。
- 退出消息写 `exit_time`。

### 9.2 同步

- 本地首次全量同步成功。
- 后续增量同步只追加或更新原始层和机器解析层。
- 本地重复同步不会覆盖人工修正层。
- 同步失败会记录错误并可重试。

### 9.3 待确认

- 待确认未处理前不影响大屏、排行榜和成员统计。
- 一条待确认原始消息可以拆成多条统计明细。
- 拆分后的多条统计明细立即影响大屏、排行榜和成员溯源。
- 原始消息仍保留为凭据。
- 忽略后的原始消息不参与统计。

### 9.4 远程用户

- 只显示数据大屏和群成员溯源。
- 不显示扫描、标签管理、发售群列表、待确认记录。
- 只能调用只读接口。
- 导出结果来自最终统计视图。

### 9.5 远程管理

- 可以访问四个页面。
- 可以管理标签和群绑定。
- 可以处理待确认记录。
- 可以查看审计日志。
- 管理员修复结果在本地和远程看到一致。

### 9.6 部署

- Vercel 部署不依赖本地文件系统。
- 远程数据库连接正常。
- API 鉴权生效。
- 普通用户 token 不能调用管理员接口。
- 管理员 token 可以调用管理接口。

## 10. 验收标准

- 三端数据一致：本地管理端、远程管理端、远程用户端读取同一远程数据库。
- 原始消息不直接参与统计。
- 最终统计视图是唯一展示口径。
- 待确认拆分、多条统计明细和人工修复规则生效。
- 本地增量扫描游标基于最大 `created_time`。
- 全局扫描从本地端移除。
- 远程应用可以部署在 Vercel 或类似平台。
