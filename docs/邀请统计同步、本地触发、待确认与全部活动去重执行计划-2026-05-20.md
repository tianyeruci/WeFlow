4. “添加信息”按钮是只删待确认列表里的那个按钮，还是连“来源候选人”快捷补充入口也一起去掉？
5. “全部活动”和“是否去重”这两个能力，是否都只做在 `WeFlow-main` 数据大屏里，还是 `WeFlow-Web` 也要同步支持同口径展示？

## 执行计划

1. 先确认接口与口径
   - 目标：把“本地刷新”接口、鉴权、CORS、全部活动范围、去重定义确认清楚。
   - 验证：确认后再改代码，避免把本机接口暴露过宽或统计口径做错。

2. 调整本地同步链路
   - 目标：把手动“同步本地数据”改成可触发完整同步；保留后台每 3 分钟的自动同步为增量/脏数据同步。
   - 目标：新增一个本机 HTTP 入口，让 Web 页点击“刷新”时能触发这条同步链路。
   - 目标：同步失败时返回明确错误，成功时返回本次同步数量。
   - 验证：本地点按钮后能看到远端数据变化，接口返回值和日志一致。

3. 调整“检查是否退出群”的调度
   - 目标：首次启动后第 1 分钟执行一次，之后每 30 分钟执行一次。
   - 目标：手动按钮不改，仍然是即时触发。
   - 验证：启动日志能看到首次与后续间隔符合预期。

4. 修正待确认记录流程
   - 目标：在“被邀请人”字段旁增加复制按钮，复制昵称到剪贴板。
   - 目标：去掉重复的“添加信息”入口，保留唯一的“有效”处理路径。
   - 目标：昵称变更但微信 ID 已知的系统消息，不再被误判成待确认。
   - 目标：点击“保存为有效”时，若只是同一来源重复但本质上是同一条消息的补齐，走正常确认或幂等处理，不再直接报“不能重复统计”。
   - 验证：用一条已知 wxid 但昵称变化的系统消息做回归检查，确认不会再落入错误待确认路径。

5. 加“全部活动”和“是否去重”
   - 目标：数据大屏增加“全部活动”虚拟选项，表示当前账号下所有活动标签的群集合。
   - 目标：`全部活动` 下按 wxid 去重时，重复记录取最早时间的一条。
   - 目标：不去重时，展示邀请人邀请到的全部数据，不按状态和归因过滤。
   - 目标：`WeFlow-main` 数据大屏增加“是否去重”按钮，默认不去重。
   - 目标：`WeFlow-Web` 默认展示不去重结果，但不显示“是否去重”按钮。
   - 验证：单活动、全部活动、去重/不去重四种组合下，人数与排行口径符合预期。

6. 收尾与记录
   - 目标：补齐类型检查、必要的构建检查和最小手动验证。
   - 目标：把本次实际修改的文件和行为变化写成本地变更记录。
   - 验证：计划执行完后，留下一份可追溯的变更说明文件。
