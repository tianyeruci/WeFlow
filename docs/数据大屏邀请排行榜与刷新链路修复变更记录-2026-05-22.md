# 数据大屏邀请排行榜与刷新链路修复变更记录

## 修改内容

- 保存执行计划：
  - 新增 `docs/数据大屏邀请排行榜与刷新链路修复执行计划-2026-05-22.md`。

- 邀请排行榜口径：
  - `WeFlow-main/electron/services/inviteStatsService.ts`
    - 不去重排行榜改为只排除 `direct` 直接入群，不再按状态、归因、退群标记、邀请人是否已知过滤。
    - 去重排行榜改为仅统计 `status=confirmed`、`valid_flag=1`、未退群、且有被邀请人 wxid 的邀请行。
    - 去重排行榜按被邀请人 wxid 去重。
    - 没有明确邀请人的记录统一归入“未知来源”。
  - `WeFlow-Web/lib/invite-data.ts`
    - Web 端排行榜使用同样的邀请溯源行判断。
    - 不去重时保留无效、待确认、已退群、未知邀请人的邀请行。
    - 去重判断补充未退群与被邀请人 wxid 要求。

- Web 页面事件数据恢复：
  - `WeFlow-Web/lib/invite-data.ts`
    - `loadFinalEvents` 改为优先读取 `invite_events` / `quit_events` 并标准化为页面现有 `FinalStatEvent` 结构。
    - `final_stat_events` 继续作为补充来源，按事件 id 合并去重。
    - 兼容 `inviter_wxid`、`inviter_wx_id`、`raw_json.join_type`、头像、时间、原始内容等字段。

- 页面端刷新链路：
  - `WeFlow-Web/lib/invite-sync-requests.ts`
    - 新增 Web 刷新请求 TTL，默认 120 秒，最小 30 秒。
    - `latest` / `next` 只返回有效期内的 `requested` 请求。
    - 过期的 `requested` / `processing` 请求标记为 `expired`，避免旧请求持续触发 Main 扫描。
    - 同一来源已有未完成的新鲜请求时复用旧请求，不重复插入新请求。
  - `WeFlow-main/electron/services/inviteStatsSyncService.ts`
    - Main 领取刷新请求前后都校验 `requestedAt` 是否在有效期内。
    - 已领取但过期的请求会回写失败并跳过，不再触发后台扫描和全量同步。

## 手工核对建议

- 同一标签/群/时间范围下，不去重排行榜总邀请数应等于群成员溯源中的邀请入群溯源行数量。
- 退出群、无效归因、待确认、未知邀请人记录应进入不去重排行榜。
- 去重排行榜只统计未退群、`status=confirmed`、`valid_flag=1` 且有被邀请人 wxid 的记录，并按 wxid 去重。
- 不点击页面“刷新数据”时，Main 不应再因为旧 `web-refresh:%` 请求触发扫描。
- 点击一次“刷新数据”应只生成或复用一个有效刷新请求，并只触发一次扫描/同步流程。
