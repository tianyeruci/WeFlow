# WeFlow 邀请统计远程管理版需求文档

## 1. 背景

邀请统计已经在本地 WeFlow 中完成核心能力，包括活动标签、群绑定、扫描、邀请解析、退群解析、成员溯源和待确认处理。为了支持管理人员远程查看和维护统计结果，需要提供远程管理版。

远程管理版不是远程控制 WeFlow 主程序，也不是远程直接读取微信本地数据库。远程管理版管理的是远程数据库中的邀请统计数据。本地 WeFlow 继续负责扫描微信系统消息，并将数据同步到远程数据库。

## 2. 总体目标

- 远程管理员可以通过浏览器管理邀请统计数据。
- 本地管理员和远程管理员读写同一套远程数据库，避免数据差异。
- 远程数据库作为邀请统计唯一事实源。
- 原始微信系统消息作为凭据永久保留，不直接参与统计。
- 一条待确认原始消息可以被人工修复为多条统计明细。
- 管理员修复结果不能被本地后续扫描同步覆盖。

## 3. 架构说明

```text
本地 WeFlow
  -> 读取微信本地数据库
  -> 增量扫描活动标签下所有群系统消息
  -> 上传原始消息和机器解析结果
  -> 作为本地管理客户端读写远程 API

远程 API / Web 应用
  -> 部署在 Vercel 或类似平台
  -> 提供管理员页面和管理接口
  -> 不直接读取微信本地数据

远程数据库
  -> Supabase / Neon / Vercel Postgres 等
  -> 保存原始消息、统计明细、人工修正、标签、群绑定、扫描日志和审计日志
  -> 作为唯一事实源
```

本地邀请统计管理允许要求联网。断网时本地可以缓存扫描得到的原始消息，但不允许执行会修改远程统计结果的管理操作。

## 4. 页面范围

远程管理版保留完整邀请统计页面：

- 数据大屏
- 发售群列表
- 群成员溯源
- 待确认记录

整体布局沿用当前邀请统计页面。远程管理版可以在页面中展示远程同步状态、最近扫描时间、最近同步时间和同步错误。

## 5. 管理能力

远程管理员允许：

- 查看数据大屏。
- 查看和筛选发售群列表。
- 管理活动标签。
- 管理群与活动标签绑定。
- 查看群成员溯源。
- 查看原始消息凭据。
- 处理待确认记录。
- 将一条待确认原始消息修复为一条或多条统计明细。
- 修复邀请人、被邀请人、退出人、wxid、群、时间、状态和归因。
- 忽略无效原始消息。
- 导出排行榜、成员溯源和原始凭据。
- 查看操作审计日志。

远程管理员第一版不直接读取微信本地数据库。

## 6. 扫描规则

### 6.1 本地取消全局扫描

本地端移除全局扫描功能，只保留增量扫描。

需要移除：

- 数据大屏中的全局扫描按钮。
- 发售群列表或其他页面中的全局扫描入口。
- IPC / API 中对 `full` 扫描模式的公开入口。
- 类型定义中的公开 `full` 扫描参数。

### 6.2 增量扫描

增量扫描扫描所选活动标签下所有已绑定且启用的群。

对每个群单独计算扫描起点：

```text
scope = activity_tag_id + group_id
last_created_time = max(raw_group_system_messages.created_time where scope)
```

如果该群没有历史原始消息：

- 扫描该群所有可读取系统消息。

如果该群已有历史原始消息：

- 只扫描 `created_time > last_created_time` 的系统消息。

只保存包含邀请或退出群聊含义的系统消息。

邀请消息：

- `event_type = invite`
- `invite_time = created_time`
- `exit_time = null`

退出消息：

- `event_type = exit`
- `exit_time = created_time`
- `invite_time = null`

扫描游标必须基于原始消息最大 `created_time`，不能只基于 `invite_time`。这样只有退出记录、没有邀请记录的群也不会重复从头扫描。

## 7. 数据模型

### 7.1 原始消息表 `raw_group_system_messages`

原始消息表保存微信系统消息凭据，不直接参与统计。

建议字段：

| 字段 | 说明 |
| --- | --- |
| `raw_message_id` | 远程原始消息 ID |
| `account_scope` | 微信账号或数据库范围 |
| `activity_tag_id` | 活动标签 ID |
| `group_id` | 群 ID |
| `group_name` | 群名称 |
| `source_message_id` | 微信消息 ID |
| `source_local_id` | 微信本地消息 ID |
| `created_time` | 微信系统消息时间 |
| `raw_content` | 原始系统消息 |
| `parsed_content` | 清洗后的系统消息 |
| `event_hint` | `invite` / `exit` / `unknown` |
| `sync_batch_id` | 同步批次 |
| `created_at` | 入库时间 |
| `updated_at` | 更新时间 |

约束建议：

- 优先唯一键：`account_scope + activity_tag_id + group_id + source_message_id`
- 兜底唯一键：`account_scope + activity_tag_id + group_id + source_local_id + created_time`

### 7.2 统计明细表 `stat_events`

统计明细表保存最终可参与统计的邀请或退出明细。一条原始消息可以生成 0 条、1 条或多条统计明细。

建议字段：

| 字段 | 说明 |
| --- | --- |
| `event_id` | 统计明细 ID |
| `raw_message_id` | 对应原始消息 |
| `source_type` | `machine` / `manual` |
| `event_type` | `invite` / `exit` |
| `activity_tag_id` | 活动标签 ID |
| `group_id` | 群 ID |
| `group_name` | 群名称 |
| `user` | 被邀请人或退出人 |
| `wx_id` | 被邀请人或退出人 wxid |
| `inviter` | 邀请人 |
| `inviter_wx_id` | 邀请人 wxid |
| `operator` | 移出群操作者 |
| `operator_wx_id` | 操作者 wxid |
| `invite_time` | 邀请时间，非邀请为空 |
| `exit_time` | 退出时间，非退出为空 |
| `join_type` | `invite` / `qrcode` / `direct` / `unknown` |
| `exit_type` | `self_exit` / `removed` / `unknown` |
| `status` | `confirmed` / `pending` / `ignored` |
| `valid_flag` | `1` 有效，`-1` 无效 |
| `delete_flag` | `1` 已退群，`-1` 未退群 |
| `created_at` | 创建时间 |
| `updated_at` | 更新时间 |

### 7.3 人工修正表 `event_overrides`

人工修正表保存管理员对原始消息或统计明细的最终解释，优先级高于机器解析。

建议字段：

| 字段 | 说明 |
| --- | --- |
| `override_id` | 修正 ID |
| `raw_message_id` | 对应原始消息 |
| `action` | `confirm` / `split` / `ignore` / `replace` |
| `operator_user_id` | 管理员 ID |
| `reason` | 修正原因 |
| `created_at` | 创建时间 |
| `updated_at` | 更新时间 |

当一条原始消息被拆分为多条统计明细时，多条 `source_type = manual` 的 `stat_events` 关联同一个 `raw_message_id`。

### 7.4 最终统计视图 `final_stat_events`

所有大屏、排行榜、成员溯源和导出统一读取最终统计视图。

规则：

- 如果 `raw_message_id` 存在人工修正结果，使用人工生成的 `manual` 统计明细。
- 如果不存在人工修正结果，使用机器确认的 `machine` 统计明细。
- `pending` 不参与统计。
- `ignored` 不参与统计。
- 原始消息本身不直接参与统计。

## 8. 待确认处理

待确认对象是原始系统消息，不是单条统计结果。

场景：

- 一条系统消息包含多个邀请关系。
- 系统消息无法唯一识别 wxid。
- 系统消息无法判断邀请人和被邀请人关系。
- 机器解析结果明显不完整。

处理规则：

- 待确认未处理前，原始消息只作为凭据保存，不参与统计。
- 管理员可以将一条原始消息拆成多条统计明细。
- 管理员可以忽略一条原始消息。
- 管理员可以替换机器解析出的统计明细。
- 修复完成后，数据大屏、邀请排行榜、群成员溯源立即按最终统计视图更新。
- 本地后续同步同一原始消息时，不得覆盖人工修正结果。

## 9. 同步规则

本地 WeFlow 扫描后主动调用远程 API。

同步内容：

- 活动标签。
- 群绑定。
- 原始系统消息。
- 机器解析统计明细。
- 扫描日志。
- 同步日志。

同步原则：

- 原始消息使用 upsert。
- 机器解析明细可以更新，但不得覆盖人工修正明细。
- 人工修正表只由管理端写入。
- 本地管理员和远程管理员都通过远程 API 写人工修正层。
- 远程数据库是唯一事实源。

## 10. 远程扫描任务

第一版远程管理员不直接扫描微信。

后续增强可以增加 `scan_jobs`：

- 远程管理员创建增量扫描任务。
- 本地 WeFlow 定时轮询任务。
- 本地 WeFlow 执行扫描。
- 本地 WeFlow 上传结果。
- 远程端展示任务状态。

由于本地端已取消全局扫描，远程扫描任务也只允许增量扫描。

## 11. 安全与审计

远程管理版必须具备：

- 管理员登录。
- 管理员 API 鉴权。
- 操作日志。
- 待确认修复日志。
- 导出日志。
- 同步日志。

关键操作需要记录：

- 活动标签新增、编辑、删除。
- 群标签绑定修改。
- 待确认确认、拆分、忽略。
- 人工修正统计明细。
- 导出。
- 远程扫描任务创建。

## 12. 验收标准

- 远程管理员可以访问完整四个页面。
- 本地管理员和远程管理员看到同一套远程数据库结果。
- 一条待确认原始消息可以修复为多条统计明细。
- 原始消息保留为凭据，不直接参与统计。
- 修复后的多条统计明细会影响大屏、排行榜和成员溯源。
- 本地重复扫描同步不会覆盖人工修正结果。
- 本地端不再存在全局扫描入口。
- 增量扫描按 `activity_tag_id + group_id` 的最大 `created_time` 继续扫描。
- 远程应用可以部署在 Vercel 或类似平台。
