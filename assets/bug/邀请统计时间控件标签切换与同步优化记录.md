# 邀请统计时间控件、标签切换与同步优化记录

记录时间：2026-05-18

## 背景

- 排行榜和群成员溯源的时间筛选使用两个原生 `datetime-local`，页面会显示较长的日期占位，布局显得拥挤。
- 发售群列表中，群从活动标签 A 切换到活动标签 B 时，不希望通过删除 A 标签数据再重扫 B 标签来实现，避免本地性能和远端 Supabase 同步压力。
- 远端同步需要从全量快照式 POST 逐步调整为 dirty 增量同步，减少重复写入和数据不一致风险。

## 本次执行中的调整

- 新增本页轻量 `CompactDateTimeRange` 时间段控件，不引入新依赖。
- 排行榜、群成员溯源改用紧凑弹层式时间范围控件：页面只显示短格式，点击后展开开始/结束秒级时间选择。
- 新增本地 `rawEvents` 原始事件缓存设计，用于保存群系统消息解析后的邀请/退群原始事件。
- `rawEvents` 不绑定 `activity_tag_id`，按 `group_id + source_message_id/source_local_id + event_type + member` 生成稳定去重键。
- 群切换活动标签时只更新 `groupTagBindings`，并迁移已有物化 `inviteEvents/quitEvents` 的标签归属，避免删除原始缓存。
- 统计读取时同时校验事件标签和当前群绑定，保证群切走后原标签不再统计该群。
- 增量扫描游标调整为按群读取 `rawEvents` 最大 `created_time/source_create_time`，避免只存在退群消息时重复从旧邀请时间扫描。
- 同步字段补充 `sync_status / sync_error / last_sync_at`，新增或修改数据标记为 `dirty`，为 Supabase 增量同步做准备。

## 后续验证重点

- 时间筛选在页面上显示为短格式，不再直接露出“年/月/日 --:--:--”长占位。
- 选择时间后排行榜和群成员溯源自动刷新，导出沿用当前条件。
- 群从 A 标签切到 B 标签后，A 不再统计该群，B 能复用该群已有原始事件。
- 反复切换标签不会删除原始缓存，也不会产生重复原始事件。
- 扫描成功后只同步 dirty 或失败重试数据，避免全量重复推送。
