# 邀请统计标签管理与溯源状态修复

更新时间：2026-05-17

## 问题 1：活动标签管理入口不符合预期

现象：
- 顶部活动标签区域曾同时提供选择、编辑、删除和常驻的新建输入框。
- 用户预期不需要在大屏和溯源页里编辑活动标签，只需要在合适的管理页里创建和删除。
- 数据大屏、群成员溯源、待确认记录顶部只需要保留活动标签选择和扫描相关操作。

解决方案：
- 移除活动标签编辑按钮和对应前端编辑逻辑。
- 保留删除能力，但只在“发售群列表”视图展示删除按钮。
- 新建活动标签改为点击“创建活动标签”按钮后弹出输入对话框，确认后创建，取消则不执行任何操作。
- 数据大屏、群成员溯源、待确认记录的顶部工具栏只保留活动标签选择和扫描按钮。

## 问题 2：顶部工具栏布局不协调

现象：
- 常驻输入框占用了过多横向空间，宽屏下视觉重心不稳定。
- 编辑、删除、创建同时出现时，工具栏层级不清晰。

优化方案：
- 将默认工具栏收敛为“活动标签选择 + 增量扫描 + 全局扫描”的紧凑布局。
- “发售群列表”增加轻量管理区，仅展示删除按钮和“创建活动标签”按钮。
- 固定扫描按钮宽度和高度，让不同视图之间的工具栏节奏保持一致。
- 移除常驻新建输入框，降低顶部信息噪音。

## 问题 3：群成员溯源状态语义不正确

现象：
- 表格状态列展示的是解析确认状态，例如“已确认 / 待确认”。
- 用户实际需要的是成员业务状态，例如“未退群 / 已退群 / 待确认”。
- 状态筛选和表格展示语义不一致，容易误解。

解决方案：
- 表格状态列改为业务状态：
  - `status !== confirmed`：显示“待确认”
  - `event_type = quit` 或 `delete_flag = 1`：显示“已退群”
  - 其他已确认入群记录：显示“未退群”
- 后端状态筛选同步按同一业务语义过滤。
- 成员溯源导出中的状态字段也同步使用“未退群 / 已退群 / 待确认”。

## 问题 4：原始消息展示不完整

现象：
- 原始消息列使用单行省略，较长的系统消息无法完整查看。

优化方案：
- 表格中改为两行预览，避免行高过大。
- 保留 `title` 悬停提示。
- 增加“查看”操作，点击或双击原始消息后弹出完整消息内容。
- 弹窗中使用自动换行和滚动区域，便于查看长系统消息。

## 问题 5：创建活动标签点击无效

现象：
- 点击“创建活动标签”后没有弹出输入框，用户无法录入新标签名称。

原因：
- 前端创建逻辑依赖 `window.prompt`，在当前 Electron 渲染环境中表现不稳定，导致点击后看起来没有响应。

解决方案：
- 改为页面内自定义“创建活动标签”弹窗。
- 点击“创建活动标签”后弹出输入对话框，输入名称后点击“确认创建”才调用 `saveActivityTag`。
- 点击“取消”、关闭按钮或遮罩时只关闭弹窗，不创建标签。
- 创建过程中禁用输入框和提交按钮，避免重复提交。

## 问题 6：本地扫描规则需要先调整，避免远程版继承错误游标

现状：
- 当前本地邀请统计的增量扫描，仍然按 `activity_tag_id + group_id` 下最后一条邀请记录的 `invite_time` 作为起点。
- 这个规则没有覆盖“只有退群消息、没有邀请消息”的场景，容易导致某些群重复从过早时间开始扫描。
- 当前页面仍保留“全局扫描”入口，不适合作为后续远程版的稳定扫描模型。

结论：
- 这一项建议在远程版开发前先改。
- 这不是单纯的页面问题，而是本地统计底座问题；如果不先修正，后续远程同步会把不稳定的扫描边界一起带过去。

调整方案：
- 移除本地端“全局扫描”能力，只保留“增量扫描”。
- 增量扫描仍然扫描当前活动标签下的全部已绑定群，但扫描起点改为按“原始消息最大 `created_time`”计算，而不是按最后一条 `invite_time` 计算。
- 增量扫描规则如下：
  - 按 `activity_tag_id + group_id` 查询当前群在原始记录中的历史数据。
  - 如果没有任何历史数据，则扫描该群全部可读取系统消息，并只保留邀请或退群相关系统消息。
  - 如果已经有历史数据，则从该群历史数据中取最大的 `created_time` 作为扫描起点。
  - 新一轮扫描只处理 `created_time >= lastCreatedTime` 的系统消息，并继续依赖现有消息级去重，避免同秒消息遗漏。

字段口径调整：
- 原始记录统一增加或补齐以下时间字段：
  - `created_time`：系统消息原始时间，始终有值，专门用于扫描游标。
  - `invite_time`：仅当系统消息为邀请或入群时写入，其他情况为空。
  - `exit_time`：仅当系统消息为退出或被移出群聊时写入，其他情况为空。
- 如果当前实现仍维持 `inviteEvents` 和 `quitEvents` 两个集合，可先兼容实现：
  - `inviteEvents.created_time = source_create_time`
  - `quitEvents.created_time = source_create_time`
  - 增量游标改为同时取这两个集合中当前 `activity_tag_id + group_id` 范围内最大的 `created_time`。

补充优化建议：
- 继续保留消息级去重，优先使用：
  - `group_id + source_message_id`
  - `group_id + source_local_id + created_time`
- 扫描结果只落邀请 / 退群相关系统消息，不把全部 `localType = 10000` 的消息都写入原始记录。
- `groupTagBindings` 中现有 `last_invite_time` 字段语义已不完整，建议后续升级为更通用的 `last_created_time`，与新的增量游标口径一致。
- 后台自动扫描也统一走新的增量规则，不再保留任何 `full` 模式入口。

验证重点：
- 某个群只有退群消息、没有邀请消息时，后续增量扫描不会重复从头扫描。
- 某个群首次扫描时，会读取该群全部可读取系统消息，但只入库邀请 / 退群相关消息。
- 同一秒内多条邀请或退群消息不会因为游标边界被漏掉。
- 去除“全局扫描”后，页面、IPC 和本地扫描服务仍保持一致。

## 问题 7：Supabase 远端同步权限不足，service_role 无法写入邀请统计新表

现象：
- 本地邀请统计数据已经成功读取，并开始通过 Supabase REST 接口同步到远端数据库。
- 远端表结构已经初始化，但同步在第一张业务表写入时失败。
- 实际错误为：
  - `permission denied for table scan_logs`
  - 提示需要执行：`GRANT SELECT, DELETE ON public.scan_logs TO service_role`

原因：
- 远端邀请统计相关表虽然已经创建完成，但没有同步授予 `service_role` 对这些表的写权限。
- 本地同步链路使用的是 Supabase HTTPS REST 写入，实际执行身份是 `service_role`；如果该角色没有 `SELECT / INSERT / UPDATE / DELETE` 权限，请求会在数据库层被直接拒绝。

影响：
- 本地到远端数据库的首批全量同步无法完成。
- 后续增量同步也会被同样的问题阻塞。
- 远端页面即使已经建表，也拿不到本地最新邀请统计数据。

解决方案：
- 在 Supabase SQL Editor 中执行授权脚本：
  - `D:\work\WeFlow\supabase\grant_invite_stats_service_role.sql`
- 该脚本会补齐以下对象对 `service_role` 的权限：
  - `activity_tags`
  - `group_tag_bindings`
  - `invite_events`
  - `quit_events`
  - `member_identity_bindings`
  - `scan_logs`
  - `final_stat_events`（只读）
- 同时已将同样的授权语句补充进：
  - `D:\work\WeFlow\supabase\init_invite_stats.sql`
  方便后续新环境初始化时一步到位。

验证重点：
- 执行授权脚本后，重新调用同步逻辑，不再出现 `permission denied for table ...`。
- 同步完成后，远端 6 张表能看到当前本地账号作用域的数据：
  - 活动标签 2 条
  - 群绑定 2 条
  - 入群事件 9 条
  - 退群事件 0 条
  - 身份绑定 8 条
  - 扫描日志 80 条

## 问题 8：邀请统计页面脱离主框架，全局扫描与指标展示不符合当前预期

现象：
- `/invite-stats` 使用独立全屏外壳，进入页面后隐藏了 WeFlow 左侧导航栏，导致无法通过左侧导航切换到其他页面。
- 顶部工具栏仍展示“全局扫描”，该入口会清理当前活动标签相关历史痕迹，当前本地端不再需要提供。
- 数据大屏第五个指标卡展示“待确认记录”，但当前预期应展示“今日退群”人数。

修复方案：
- 恢复邀请统计页使用 WeFlow 主窗口布局，保留左侧 `Sidebar` 和标题栏折叠按钮，不再对 `/invite-stats` 使用独立 shell。
- 删除页面上的“全局扫描”入口，并收窄前端、preload、IPC、服务层的扫描调用为增量扫描。
- 数据大屏第五个指标卡改为“今日退群”，统计当前活动标签下北京时间 0 点以来已确认退群事件的去重人数。

验证重点：
- 进入邀请统计页后左侧导航栏可见，点击其他导航项可以正常切换页面。
- 邀请统计所有页顶部只保留“增量扫描”，不再出现“全局扫描”按钮或确认弹窗。
- 数据大屏第五张指标卡标题为“今日退群”，数值随当天退群记录变化。
