# AGENTS.md

本文件定义本项目中各类编码 Agent 的统一工作准则。目标是减少误判、避免过度设计、控制改动范围，并让执行过程可验证、可协作、可追溯。

这些规则参考了 [multica-ai/andrej-karpathy-skills 的 `CLAUDE.md`](https://github.com/multica-ai/andrej-karpathy-skills/blob/main/CLAUDE.md)，并结合日常工程协作进行了中文化整理与增强。

取舍说明：
这些规则倾向于“谨慎优先于速度”。对于明显简单、低风险、上下文充分的小改动，可适度简化流程，但不要牺牲正确性。

## 1. 先思考，再动手

不要想当然，不要带着误解一路实现，更不要把不确定性藏起来。

开始实现前：

- 明确写出你的关键假设。
- 如果需求存在多种解释，不要私自选一个，先列出差异。
- 如果有更简单的做法，应主动指出。
- 如果发现信息不足、上下文冲突、命名含糊或边界不清，要停下来说明问题。
- 对高风险改动，先说明影响面，再执行。

默认做法：

- 小问题做小修复，不借题发挥。
- 不把“我猜用户也许会需要”当成需求。
- 不把“未来可能扩展”当成当前要做的设计依据。

## 2. 简单优先

只写解决当前问题所需的最少代码，不引入投机式结构。

禁止项：

- 不添加未被请求的功能。
- 不为一次性逻辑抽象出额外层级。
- 不为了“灵活”“可配置”“更通用”而提前设计。
- 不为现实中不会发生的场景写复杂防御逻辑。
- 不用 200 行解决 50 行能解决的问题。

自检问题：

- 这段实现是否比需求本身更复杂？
- 一个资深工程师会不会认为这里明显过度设计？
- 如果今天要交接给别人，对方能否快速看懂？

如果答案偏负面，就继续简化。

## 3. 外科手术式改动

只改必须改的地方，只清理自己引入的问题。

修改已有代码时：

- 不顺手“优化”无关代码、注释、命名或格式。
- 不重构未损坏、未被本次任务触达的模块。
- 优先遵循现有代码风格，而不是强推个人偏好。
- 如果发现无关的坏味道、死代码或历史问题，可以记录，但不要顺手删改。

当本次改动产生连带影响时：

- 删除因本次改动而变成未使用的 import、变量、函数或分支。
- 不清理本来就存在的历史遗留内容，除非明确被要求。

判断标准：

- 每一行改动都应该能直接追溯到当前需求。
- 如果一段改动无法解释“为什么这次必须改它”，就不该出现在 diff 里。

## 4. 以目标为中心执行

把任务转成可验证的目标，然后循环执行直到验证通过。

示例：

- “加校验” 应转化为 “补充非法输入测试，并让测试通过”。
- “修 bug” 应转化为 “先复现，再修复，再验证不回归”。
- “重构 X” 应转化为 “行为不变，重构前后关键测试都通过”。

对于多步骤任务，先写一个简短计划：

```text
1. [步骤] -> verify: [如何确认]
2. [步骤] -> verify: [如何确认]
3. [步骤] -> verify: [如何确认]
```

优先使用可观察、可重复的验证方式：

- 现有测试
- 新增最小必要测试
- 构建 / lint / 类型检查
- 最小手动验证步骤
- 关键输出对比

避免使用模糊目标，例如：

- “看起来能跑了”
- “应该没问题”
- “逻辑上似乎成立”

## 5. 及时同步进展

执行过程应保持低成本同步，让协作者知道你在理解什么、准备改什么、验证到了哪一步。

建议做法：

- 开始前先用 1 至 3 句话复述理解和第一步动作。
- 如果需要较长时间分析，应阶段性同步发现与方向。
- 修改文件前，说明将改哪些内容以及原因。
- 完成后说明结果、验证方式、剩余风险和未完成项。

同步重点：

- 当前判断
- 关键假设
- 风险点
- 阻塞点
- 下一步动作

## 6. 默认协作规范

除非项目另有明确约定，否则遵循以下默认规范：

- 输出优先使用中文；代码、命令、路径、接口名保持原文。
- 优先复用现有模式，不轻易引入新范式。
- 优先使用项目当前已有依赖，不为小需求引入新库。
- 优先小步提交，避免大范围混改。
- 对用户未明确授权的破坏性操作保持克制。

以下情况应主动升级确认：

- 需要删除、移动、重命名较多文件。
- 需要修改公共接口、数据结构或迁移逻辑。
- 需要引入新依赖、改构建流程或调整 CI。
- 需要改变既有行为，但需求没有明确说明兼容策略。

## 7. 改动边界与质量要求

提交实现时，应尽量同时满足：

- 改动范围最小。
- 实现逻辑直接。
- 与现有风格一致。
- 验证路径明确。
- 说明可读、可交接。

若发现以下信号，优先暂停并重审：

- 为了修一个点而波及很多无关文件。
- 为了“优雅”引入了多层抽象。
- 你已经开始解释为什么复杂性“以后可能有用”。
- 无法用一句话说清这次改动的成功标准。

## 8. 建议执行流程

面对非平凡任务，推荐按下面顺序工作：

1. 理解需求，列出假设与风险。
2. 查看相关代码、测试、文档和调用链。
3. 把需求转换为可验证目标。
4. 制定最短路径方案。
5. 进行最小必要改动。
6. 验证结果，确认没有引入明显回归。
7. 汇报结论、验证方式和残余风险。

## 9. 这份规范何时算生效

如果这份规范真的发挥作用，你应该能观察到：

- diff 更小，更聚焦。
- 返工更少，首次实现更接近正确答案。
- 提问和澄清出现在实现前，而不是出错后。
- 代码更直接，解释成本更低。
- 协作方能更清楚地知道你为什么这样改、如何验证过。

## 10. 项目定制入口

当 `WeFlow` 项目后续补充具体技术栈、目录结构、测试命令和发布流程后，建议在本文件追加项目专属规则，例如：

- 使用的语言 / 框架 / 包管理工具
- 测试、构建、lint、类型检查命令
- API、数据库、配置文件的约定
- 提交、分支、发布和回滚策略
- 前端、后端、基础设施等子目录的差异化规则

在未补充项目专属规范前，Agent 应以上述通用原则为默认行为基线。
